# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A **static, client-side mockup** of the Fairway Canyon HOA Architectural Review Committee (ARC) application, rebuilt from a printed PDF (`docs/source-form.pdf`) as a web form for GitHub Pages. **There is no backend.** Nothing is transmitted: drafts persist to `localStorage`, and the journey **ends at a printable packet** — the user saves/prints a self-contained PDF (submission instructions + application + neighbor signature form) and submits it themselves; data can also be exported as JSON. (An earlier `mailto:` email flow was removed 2026-07-05 — the print packet's page 1 carries the submission instructions instead.)

Plain HTML/CSS/JS — **no build step, no npm dependencies, no linter.** The behavior is native **ES modules** under `assets/` (loaded directly by the browser via `<script type="module">`); the pure geometry module has a **`node --test` unit suite** in `tests/`, and the repo-root `package.json` exists *only* to set `"type": "module"` for Node tooling (no deps, no scripts beyond `test`). MapLibre GL JS (maps) and Konva.js (the plot-plan drawing stage — a raster material grid plus vector annotations) are **vendored as exact pinned builds in `assets/vendor/`** (version in the filename; to upgrade, download the new dist file there, update the `index.html` tags, and commit both together — never reintroduce a floating CDN tag). The one remaining CDN dependency is Google Fonts, which degrades gracefully to system fonts. (There are two dev-time Node scripts, `tools/fetch-streets.mjs` and `tools/fetch-parcels.mjs`, but they are not part of the app or its build — see Commands.)

## Commands

- **Run locally:** `python -m http.server 8000`, then visit `http://localhost:8000`. Use a server rather than opening `index.html` via `file://` — ES modules won't load at all under `file://` (CORS), and the county GIS `fetch()` calls and map tiles need an http(s) origin anyway.
- **Deploy:** GitHub Pages serves the repo root (`.nojekyll` disables Jekyll processing). The live branch is **`master`**.
- **Tests:** `node --test` (or `npm test`) runs `tests/geometry.test.js` — 37 cases over the pure geometry (grid sizing/clamps, similarity fit, auto-align, orientation snap angles, flood-fill walls/coverage rules, point-in-polygon). Run it after touching `assets/geometry.js`, and extend it when moving more pure logic there.
- **Sanity check:** always run `node --check` on every `assets/*.js` file you edited — it catches syntax errors before you hand a change to the user (who does the real browser verification). No build step: edit the source files and reload.
- **Regenerate the baked street extract:** `node tools/fetch-streets.mjs` rewrites `assets/streets.json` from live Overpass (it tries several public mirrors in order). Widen the `BBOX` in that script if the service area grows. Dev-time tool only — the deployed app just `fetch()`es the committed JSON.
- **Regenerate the baked parcel extract:** `node tools/fetch-parcels.mjs` rewrites `assets/parcels.json` (~3.7 MB raw / ~650 KB gzipped, ~7.5k parcels) from the county assessor layer via paged queries. Keep its `BBOX` identical to fetch-streets'. It prints the community center of mass — if it moved, sync the `HOA_CENTER` fallback constant in `assets/map-wizard.js`.

### Verification / testing this agent CAN'T do well

**Ask the user to test map-wizard and drawing-surface changes for you — don't burn turns trying to drive them via browser automation.** The map wizard (Select → Orient → Draw) and the Konva plot surface are effectively un-testable from Claude Code's browser automation in this environment, for concrete reasons learned the hard way:
- The MapLibre map frequently sits **below the fold and won't scroll into the clickable viewport**, so screen-clicking a parcel is unreliable.
- The dev server (`python -m http.server`) sends no cache headers, so the browser **serves a stale `assets/app.js` even after edits + reload** — you have to cache-bust the `<script src>` to load fresh code, which is throwaway churn.
- Driving the flow programmatically needs a **temporary `window.__dbg` hook** (module internals aren't otherwise reachable), plus the county GIS lookup, the auto-align camera animation, and the aerial-backdrop export fetch all have real latency/timing that browser automation races against.
- Screenshots intermittently time out or render faded/blank in this sandbox.

So: implement the change, `node --check` the edited modules (+ `node --test` if geometry changed), reason carefully about the geometry/timing, then **hand it to the user to verify in their own browser** (they have the real map, real parcels, and can eyeball scale/alignment instantly). Reserve heavy browser-automation verification for the static form sections that actually sit in the viewport — one workaround for the stale-cache problem there: serve on a **fresh port** for each verification round.

## Architecture

Markup is `index.html`; presentation is `assets/styles.css` (~1200 lines, organized in clearly-labeled `/* === SECTION === */` blocks ending with a `@media print` block that styles the print output). Behavior is **six ES modules under `assets/`**, loaded natively (entry: `<script type="module" src="assets/app.js">`; the vendored Konva/MapLibre stay classic scripts that define globals):

- **`utils.js`** — `$`, `$$`, `esc`. Imported everywhere.
- **`dropzone.js`** — drag-and-drop + clipboard-paste attachment behavior (a leaf module, no app imports). `registerDropzone(zoneEl, input)` makes any element a drop target for its `<input type=file>`: accepted files land via `DataTransfer` and dispatched bubbling `input`+`change` events, so the existing status/progress/autosave listeners run unchanged. Honors the input's `accept` (mismatches get a reject-shake), **appends** on `multiple` inputs (the multi-file close-up shot), replaces on single. Paste routes to the hovered zone → the zone owning focus → the single zone in the viewport (else deliberately dropped — guessing would misfile the image). Registered zones: each `.photo-request` card (the whole card, so a drop still replaces once `.has-thumb` hides the frame), each improvement row's frame, `#plot-upload`. The `.dropzone` frames keep the real input in-DOM, visually hidden but focusable (`.dropzone__input`), so the upload path's keyboard completeness survives (`:focus-within` ring, Enter opens the picker); `body.is-file-drag` (window-level dragenter counter) lights all zones during any file drag, and document-level `dragover`/`drop` preventDefault stops the browser navigating to a file dropped outside a zone.
- **`geometry.js`** — the pure, DOM-free math: scale constants (`CELL_SIZE`, `FOOT_IN_METERS`, `GRID_*`, `MAX_GRID_DIM`), parcel projection (`geoToLocalMeters`/`rotatePoints`/`computeBBox`/`buildParcelGrid`), `fitSimilarity`, `computeAutoAlignBearing(ring, roadDir)` (road direction is a **parameter**, not module state), `computeSnapAngles(ring)`/`snapBearing(deg, angles, tolerance)` (the Orient step's snap-to-lot-line candidates — each boundary edge's grid-squaring bearing collapsed mod 90°, short cul-de-sac arc segments filtered out), `closestPointOnSegment`, `pointInPolygon` (the Stamp tool's parcel-boundary gate — re-added with a real caller after an earlier dead copy was deleted in Sprint 3), and the flood-fill core (`segmentsIntersect`, `connectorBlocked`, `pointOnSegment`/`pointOnAnyWall`, `computeFloodFill`). **Unit-tested** (`tests/geometry.test.js`); keep it dependency-free and put new pure logic here.
- **`plot-editor.js`** — the Konva drawing surface (everything in "Plot drawing surface" below) plus its data constants `PALETTE`/`RETIRED_MATERIALS`/`STAMPS`/`ICON`/`TOOL_MODES`. Owns the grid/canvas/undo state, the custom-materials list, and the **explicit-completion flag** (`isPlotConfirmed()`/`setPlotConfirmed(v)` — see "Draw-step chrome" below). Cross-module API: `rebuildGridForParcel(feature, bearing, apn)`, `setPlotBackdrop(dataUrl, parcelPx)` (the wizard's aerial handoff — an async county exportImage fetch, so it usually lands after the grid is already built), `plotUsed()`, `isPlotConfirmed()`/`setPlotConfirmed(v)`, `renderPlotImage()`, `plotLegend()` (materials painted + stamps placed, feeding the preview/print legend), `serializePlot()`/`restorePlot(plot)` (persistence handshake).
- **`map-wizard.js`** — the Site/Plot Plan 4-step flow, MapLibre map, county-GIS lookups, rotation, plan-mode choice, OSM street lookup. Exports live-binding state the form reads (`planMode`, `parcelBearing`, `selectedAPN`, `selectedParcelGeoJSON`, `mapInstance`, `plotUploadInput`) plus `setPlanMode`, `setRotation`, `restoreParcelFromDraft(feature, apn, bearing)`.
- **`app.js`** — entry + form/persistence: `ACKS`/`DATES`/`PHOTO_SPECS` data, `SignaturePad`, photos, progress meter, packet status, `collect()`, drafts, validation, preview/print/JSON outputs, landing, init. Exports `scheduleAutosave`, `updateProgress`, `setFieldError`, `refreshPacketUI` for the other modules.

**Module-evaluation order is load-bearing**: `app.js` imports plot-editor before map-wizard, so top-level init runs plot editor → wizard → form, exactly the old single-file order. plot-editor and map-wizard import those four `app.js` functions back — a **deliberate ESM cycle** that is safe only because they're hoisted function declarations called at event time; never *read a non-function binding* from `app.js` at another module's top level (TDZ crash at load).

Content is data-driven from constants — **edit these arrays to change form content**, not the generated DOM:
- `ACKS` (app.js) — the 8 acknowledgment statements (HTML strings, rendered into `#acks`)
- `PALETTE` (plot-editor.js) — the plot-plan **area** materials (`id`/`label`/`color` + optional `texture`, a key into the `TEXTURES` registry of drafting hatches — speckle/dots/diag/cross/brick/dashes/tufts — baked into the raster paint via canvas patterns so print/preview get them for free; `materialSwatchStyle(m)` is the one exported helper that renders a textured swatch background for chips and legends), rendered by `renderPaletteChips()` into the material chip row (`#palette`, the one palette above the canvas) together with the user's custom materials and the "+ Add material" chip; `RETIRED_MATERIALS` (tree/light/camera — point objects, now Stamps) keeps old ids resolvable in `PALETTE_MAP` without a chip; `STAMPS` is the list of point-object plan symbols, two kinds sharing one array — 24 full-color plant/landscape icons (`img`: a path into `assets/stamps/*.svg`, `box`: 64, the vendored plant-icon starter set) plus 3 legacy hardware symbols (Camera/Yard Light/AC Unit, `d`: a 24-box stroke-only SVG path, default `box` 24) — each with `id`/`label`/`ft` real-world footprint, rendered as the Stamp tool's glyph picker (`#stamp-control`) and drawn as `kind:"stamp"` groups; `TOOL_MODES` is the separate list of drawing-tool modes (paint · rect (default) · erase · fill · stamp · line (labeled "Outline") · callout · measure · select · pan, each paired with an inline SVG from the `ICON` map, **a single-letter hotkey `key`** (M/R/E/F/S/O/C/D/V/P, active only while the pointer hovers the canvas, surfaced in the rail tooltips) **and a one-line `hint` HTML string** — `updateToolHint()` swaps it into `#tool-hint` in the status strip *below* the canvas whenever `setActiveMode()` runs), rendered **icon-only** into `#tool-palette` inside the vertical `.tool-rail` (labels live on `data-tip`/`aria-label` and lead the hint text — `data-tip` drives the **instant CSS tooltips** on the whole draw chrome: `.tool-rail button::after` flows right from the rail, `.plot-status__stamps button::after` flows up from the stamp picker, and the Undo/Redo buttons carry `data-tip` in the markup; native `title` was dropped wherever `data-tip` applies to avoid double tooltips). Note `ICON.erase`/`ICON.fill`/`ICON.pan` are **48-box** glyphs (user-supplied, stroke-width 3.8) while the rest are 24-box at stroke 1.9 — `cursorForMode()` reads each icon's viewBox and scales the cursor halo/ink stroke widths accordingly, so don't assume 24 there
- `DATES` (app.js) — the 2026 review-date table (rendered into `#dates-body`, which lives on the **landing page**)
- `PHOTO_SPECS` / `PHOTO_MATERIAL` (app.js) — the per-area photo requests for the Photos section (each shot has `id`/`title`/`instr`/`good`/`bad`); editing these changes which photos are requested and their instructions
- `CATEGORIES` (app.js, Sprint 11) — the Section 02 improvement **types** (`id`/`label` + two per-type field slots, `materials` and `dims`, each `{label, ph}`; `dims: null` hides the dimensions field). They drive what each improvement row asks: structure/hardscape ask materials + dimensions, plants ask "plant type & quantity" + mature size, paint asks a color code with **no** dimensions, equipment asks make/model + location, etc. `applyImprovementSchema()` relabels/hides the two slots per the chosen category; `categoryLabelShort()` feeds the output table's Type column

The form is a **multi-section single page**, gated behind a **landing/cover screen**. DOM/section order (and card numbers): landing → `01` Applicant · `02` Proposed Improvements · `03` Site/Plot Plan · `04` Photos · `05` Acknowledgments · `06` Review & Submit (`#packet` — packet checklist + a **single finish step**, Save & print the packet; replaced the old sticky three-button submit bar, absorbed the old "Required Submissions" card in the 2026-07-03 journey de-dup, and the last manual checkbox — the sketches attestation — was retired in Sprint 12, so **every packet row is derived** and old drafts' `req_sketches`/`req_fee` keys restore harmlessly into nothing). **Order reflects the Sprint 10 journey rework** (Improvements sits *before* Site/Plot: name the change, then place it on the lot) **and the 2026-07-05 print-first rework**: the old Adjacent Owners roster section and the email finish steps were removed entirely — the neighbor signature form is now the **last page of the printed packet** (blank rows, wet signatures collected in person after printing), and the packet's **page 1 is submission instructions**, so the user never returns to the web form after printing. The sidenav, card numbers, and DOM order must agree when sections are added/reordered.

### Key subsystems (landing/photos/persistence/packet/validation in app.js; plot surface in plot-editor.js; wizard/geocoding in map-wizard.js)

- **Landing / form reveal** — a full-screen cover (`#landing`, between the masthead and the form) holds the unpacked required-submissions overview + the 2026 review-dates table + how-to-submit/assistance. The whole form (`.layout`, `id="form-layout"`) starts with the `hidden` attribute; `enterForm()` (wired to `#start-application`) unhides it, and `showLanding()` (wired to the sidenav `#view-landing` link) returns to it. **Returning users skip the landing:** `restoreDraft()` returns a boolean and `enterForm()` is called when a draft was restored.

- **Proposed improvements** (`#description`, Section 02) — an **item repeater** (generated rows), replacing the old single free-text proposal. `improvementTemplate(idx)`/`addImprovement()` build each `.improvement` row: a **category** `<select>` (`[data-imp-category]`, from `CATEGORIES`), an **action** `<select>` (Add / Replace / Remove), a full-width **name** (`[data-imp-name]`, `[required]` — this is where Section 02's requiredness now lives), two **flexible slots** (`[data-imp-materials-field]` / `[data-imp-dims-field]`), and a file input for an **example/catalog picture** (`[data-imp-photo=idx]`). `applyImprovementSchema(node)` (Sprint 11, generalizes the old `applyImprovementAction`) reflects **both** category and action: the **category relabels the two slots and hides the dimensions field** for types that don't need it (paint — a plant asks "Plant type & quantity", paint asks a "Color name, code & manufacturer"), while **Remove additionally hides materials** (via the `hidden` attribute — safe because `.field` sets no `display`) and relabels the picture as an optional "what's being removed" shot. (A CSS `:has()` rule lets materials span the row when the category hides dimensions.) The per-row remove button **re-adds a fresh row if it would empty the list**, so at least one `[required]` name always exists (the "at least one item" rule); `renumberImprovements()` (Sprint 12) re-derives the visible "Improvement N" labels from 1-based DOM position after every add/remove, while `data-improvement`/input ids stay on the monotonic `improvementCount` (uniqueness for the label/`for` pairs and photo-status wiring). `improvementItems()` is the single DOM→data source (`[{category,action,name,materials,dimensions,photo}]`, `photo` = the real attached filename or `""` — an un-reattached picture drops on the next save, same browser limit as photos); `improvementChecklist()` derives the packet rows (Add/Replace items with a name; Remove items need no picture). The demoted `#proposal` textarea is now an **optional** "Anything else the committee should know" notes field (no longer `[required]`; the char counter stays). Outputs render via the shared `improvementsTableHTML()` (see Output paths).
- **Photos** (`#photos-section`) — a **questionnaire drives which photos are requested**. The two questions (which areas: `area_front/back/side/exterior` checkboxes; plus a `photoMaterial` yes/no radio) toggle pre-built per-area groups via `refreshPhotoGroups()` (it flips each `.photo-group`'s `hidden`, and shows/hides the `#photo-empty` placeholder). Groups/blocks are generated once by `buildPhotoRequests()` from `PHOTO_SPECS`/`PHOTO_CLOSEUP`/`PHOTO_MATERIAL`, so already-attached files survive a questionnaire change (groups are hidden, not destroyed). **`PHOTO_CLOSEUP` (Sprint 12) is the area-agnostic "Close-up(s) of the work area(s)" shot** — its own group (`data-area="closeup"`), shown whenever **at least one** area is checked (it used to live in the back-yard group, so front-yard work never asked for it); the id stays `back_closeup` so pre-Sprint-12 drafts restore into it. It's the one **`multiple`** photo input (one file per work area — dropzone.js appends on multiple; the native picker replaces): the status line shows a count, thumbnails render one card per file into `.photo-thumb-list` (each with its own Remove, which rebuilds the FileList via `DataTransfer` — a FileList is immutable), the dropzone stays visible after attaching (no `.has-thumb`), and its `photos` map value persists as a filename **list** (restore + preview/print/manifest all accept string-or-array). Each shot has its own `<input type=file data-photo-input=ID>` + a ✓/✗ example pair — real photos (`assets/photo-examples/<id>-good.jpg`/`-bad.jpg`, EXIF-stripped) wired via optional `goodImg`/`badImg` keys on the `PHOTO_SPECS` shot objects (`exampleFrame()` renders the `<img>` when present, else the original placeholder frame); 9 of 10 shots have both, and only `material_sample` still has no source photo at all so it keeps the placeholder on both sides (ISSUES #3). `collect()` emits `photoAreas`, `photoMaterial`, and a `photos` map (`id → filename`, or a filename **list** for the multi-file close-up shot); `files` is the flat list of attached photo names for preview/print. **Attaching a live file shows a framed thumbnail** (`updatePhotoStatus()` mints a `URL.createObjectURL` blob into the `.photo-thumb` card — image preview + green ✓ badge, filename, size, and **Replace/Remove** controls — and adds `.has-thumb`, which CSS-hides the plain upload row); the id→blob-URL cache `photoThumbUrls` (an **array** per id since the multi shot mints one URL per file) is revoked before each replace and on Remove so blob handles can't leak. Replace re-opens the picker (`input.click()`); Remove clears `input.value` and — since a programmatic clear fires no change event — drives `updatePhotoStatus`/`updateProgress`/`scheduleAutosave` by hand. As with all file inputs, restore can only show the prior filename (browsers can't repopulate a file input), shown as an `is-prior` status **with no thumbnail** (the bytes are gone; re-attaching brings the thumbnail back).

- **`SignaturePad`** (the one class) — DPR-aware `<canvas>` signature capture via pointer events. Serializes to a PNG dataURL (`toDataURL`/`fromDataURL`) so signatures survive localStorage round-trips. Instances are tracked in the `sigPads` map, keyed by the element's `data-sigpad` attribute. **Hidden-safe sizing:** `resize()` no-ops while the pad is `display:none` (the landing gate, or the Type method below) so a 0×0 rect can't clobber the bitmap; `enterForm()` sizes the pads on reveal, and `fromDataURL` on a hidden pad parks the url in `this.pending` (applied on the next resize, and passed through by `toDataURL()` so a hidden-pad draft round-trip doesn't lose ink). **Signature method toggle (Sprint 5):** the owner signature offers Draw/Type (`#sig-method-draw`/`#sig-method-type` → `setSigMethod()`, state in `sigMethod`); Type reveals `#typed-signature`, whose typed full legal name is the keyboard/AT-accessible e-signature alternative. `ownerSignatureProvided()` is the method-aware "signed?" check used by validation and the progress meter; both inputs persist in the draft (`ownerSigMethod`/`ownerTypedSignature`, additive to `.v4`), and preview/print render whichever method is active (typed names get a script face, plus a "signed electronically by typing name" label in print).

- **Plot drawing surface (raster grid painter + Konva annotations)** — the "Draw Your Plan" step (`#plot-konva-host`) is a `Konva.Stage` with **four** layers, bottom-to-top: `bgLayer` (`listening:false`; aerial snapshot + dashed parcel outline), `gridLayer` (`listening:false`; two stacked `Konva.Image`s wrapping offscreen `<canvas>`es — `gridLinesCanvas` graph paper + `paintCanvas` painted materials — with `mixBlendMode:"multiply"` set on the layer's own DOM canvas so materials read *through* the aerial on screen, while `toDataURL()` ignores the blend so print stays opaque), `drawLayer` (Konva callout + measurement groups + Line-tool lines + Stamp groups **only**), and `overlayLayer` (in-progress ghost shapes, the Stamp cursor preview, the Marker/Erase **brush-footprint preview** (`updateBrushGhost` — a cell-snapped square shown whenever the dialed thickness is > 1, incl. during right-click quick-erase strokes; cleaned up on tool switch, stage `mouseleave`, pan/Space start, pinch start, and the Done-lock) + the Select-highlight rect — excluded from persistence/export just by living on its own layer). **Scale is fixed: 1 grid tile = 1 ft = `CELL_SIZE` (8) px at 100% zoom.** A bigger lot yields a *bigger grid you pan/zoom around*, never a shrink-to-fit one, so a foot is always the same size on screen. **Materials are raster**, held in the `cellState` `Map` (`"c,r" → materialId`) and drawn with real cell ops dispatched from `onStagePointerDown/Move/Up`: `paintBrush`/`paintStroke` (a square brush of `brushSize` ft, interpolated so a fast drag leaves no gaps), `fillRect` (Rectangle rubber-bands a block; **Shift** on Paint draws a straight line), and a genuine 4-way **`floodFill`** paint-bucket. Right-click / Ctrl(⌘)-click is a quick eraser from any tool — and with the **Fill** tool specifically it's a *bucket eraser*: it flood-fills the clicked region back to empty (`floodFill(..., null, ...)`) rather than erasing one cell (the `fill` branch in `onStagePointerDown` handles the erase gesture *before* the generic brush-erase branch, so it wins over the single-cell quick-erase). **Annotations are vector Konva**: callouts are a `Konva.Group` (`Label`+`Tag`+`Text` plus a free-angle `Konva.Arrow` leader, since `Tag.pointerDirection` only supports 4 cardinals) measurements a `Konva.Group` (`Konva.Arrow` + a `Konva.Text` distance label via `scaleFeetPerPixel`), and the **Line** tool a plain `Konva.Line` (`kind:"line"`) whose endpoints `snapToGrid` to the nearest grid-line intersection and whose stroke is a real-world 2 in (`gridLineWidth()` = `LINE_WIDTH_FEET / scaleFeetPerPixel`, so it stays 2 in at any zoom) with a fat `hitStrokeWidth` so the hairline is still clickable — and Lines also act as **flood-fill walls at any angle** (triangles included): `floodFill` floods at `FILL_SUBSAMPLE`² (4×4) sub-tile resolution and blocks a step whenever the connector between two subcell centres crosses a Line (`collectLineSegments`/`segmentsIntersect`/`connectorBlocked`) — watertight for straight segments since a straight connector crosses a straight segment iff its endpoints are on opposite sides, so a closed shape whose corners share snapped vertices fully contains the fill — then paints a whole tile iff **>50% of its subcells are inside** (the "more than half the tile is within bounds" rule, which also subsumes axis-aligned walls and 100%-interior tiles). Coverage = reached subcells **plus the on-wall strip** (`pointOnAnyWall`) that the conservative touch-block otherwise erodes — without this a diagonal-cut tile reads <50% and leaves a visible gap, so triangles wouldn't fill to their edge. Filling the *other* side of a wall can't overwrite already-painted edge tiles (the material-region check blocks the second flood from entering them), so the **first fill keeps the shared edge — no gap, no overwrite**. The fill is seeded from the subcell under the click so it starts on the clicked side of a wall; `attachShapeInteractions` wires Select and Erase (click-to-delete) per node — Select drags a whole shape, and selecting one also grows **round reshape handles** on `overlayLayer` (`buildEditAnchors`: outline & measurement endpoints, callout arrowhead + label box; each handle is a content-space get/set pair so it survives a prior whole-shape drag; outline endpoints re-`snapToGrid`, measurement edits re-derive the feet label via `syncMeasurementLabel`, and **double-click re-prompts a callout's text** via `editCalloutText`) — and annotations stay `listening(false)` outside those two modes (one decider: `syncNodeInteractivity`, which also reads the Done-lock and held-Space flags) so a paint stroke passes straight through to the grid beneath. **Deleting a shape is one shared path (`deleteAnnotation`) with four routes** (Sprint 13): the Erase-tool click, **right-click from any tool** (`annotationAtPointer` temporarily flips `listening` on and rebuilds the hit graph, since annotations are deliberately non-listening outside Erase/Select), **Backspace/Delete** on the current selection, and a **red "×" button** on the selection box's top-right corner (`buildSelectionDeleteBtn`, overlayLayer, counter-scaled with the edit anchors) — and a press landing directly ON a shape deletes only that shape, never the paint beneath it (the right-click hit test runs before every erase branch, and the Erase tool skips starting a cell-erase stroke when `annotationRootOf(e.target)` hits; a stroke that *starts* on empty ground still sweeps over shapes without deleting them). **Nothing draggable can leave the canvas**: whole-shape drags are clamped by a `dragBoundFunc` (`annotationDragBound`, bbox-clamped to the content rect — re-attached on every hydrate since functions don't serialize), reshape anchors are clamped to the content rect, and the callout label box gets a box-aware clamp (its anchor spec's `clampBox` + a creation-time nudge in `commitCallout`), so a label can't hang off the right/bottom edge. **Stamps (Sprint 7, plant-icon catalog added later)** are the fourth vector kind: a `kind:"stamp"` `Konva.Group` (attrs carry `stampId`) holding either a `Konva.Image` (the 24 full-color plant/landscape icons, `assets/stamps/*.svg`) or, for the 3 legacy hardware symbols, the glyph `Konva.Path` twice — a fat white halo under the black stroke, the standard plan-symbol treatment so it reads over aerial and paint alike — plus an **invisible full-footprint hit `Circle`** (`opacity:0`; Konva's hit graph ignores opacity) so Select/Erase catch clicks anywhere inside the symbol, not just on a hairline stroke. The glyph's `box`×`box` viewBox is scaled to the symbol's real-world `ft` footprint (`buildStampGroup`). **Image stamps need a rehydration step Path stamps don't**: Konva's serialized JSON can't carry an `Image` node's actual bitmap, so every path that rebuilds a stamp from JSON (`hydrateShapesInto` — draft restore, undo/redo — and `renderPlotImage`'s offscreen composite) calls `rehydrateStampImages(node)` afterward, which re-attaches the right `<img>` element from `STAMP_IMG_CACHE` (a module-level cache of preloaded, same-origin `Image()` elements keyed by stamp id, populated once at load). Placement **arms on press and commits on release** (`stampArmed`) so `beginPinch()` can cancel a two-finger tap, mirroring the paint-stroke rollback — and only lands **inside the parcel footprint** (`insideParcel` → geometry's `pointInPolygon`; outside, the ghost dims and the cursor flips to not-allowed, though Select can still drag a placed stamp out); a translucent ghost (`stampGhost`, on `overlayLayer`) previews the footprint under the cursor and is cleaned up on tool switch, stage `mouseleave`, and pinch start. Everything else — drag/erase wiring, undo, `serializePlot`/`restorePlot` (via `hydrateShapesInto`), print via `renderPlotImage()` — falls out of the shared drawLayer pipeline; `collectLineSegments()` skips them by `kind`. **Pan/zoom is DOM-scroll based**: `applyStageSize` scales the stage by `viewZoom` and the `#plot-konva-host` element scrolls; `minZoom`/`maxZoom`/`clampZoom` bound you between "no whitespace" and a `SAFE_CANVAS_PX` memory cap; `fitView` ("Fit") zooms out and centers on the parcel; the Pan tool / middle-mouse drag / **held Space** (temporary pan from any tool; `spacePan`, gated on the pointer hovering the host via `plotHover`, works on a locked plan too) adjusts native scroll. **Touch: a two-finger pinch zooms/pans** while one finger keeps using the active tool — host-level capture-phase pointer tracking (`touchPts`/`pinch`) sets the pinch flag before Konva's own `pointerdown` could start a draw with the second finger; `beginPinch()` cancels any in-progress gesture and rolls a half-painted stroke back through the undo point recorded at its start; `onStagePointerDown/Move` early-return while a pinch is live; zoom+pan is one formula (pin the content point under the fingers' starting midpoint to the current midpoint). **Undo/redo (`snapshotState`) captures BOTH** `cellState` (as pairs) **and** `drawLayer.toObject().children` onto a capped (50-deep) stack, recorded via `recordUndoPoint()` at the *start* of each gesture, not the end. `renderPlotImage()` (used by `buildPreview()`/`buildPrintHTML()`) composites a **throwaway offscreen `Konva.Stage`** — never the live one — from `gridLinesCanvas` + `paintCanvas` + the parcel outline + `drawLayer.toJSON()`, so pan/zoom state and overlay ghosts can never leak into print. Autosave (`scheduleAutosave()`) is called explicitly from every tool's commit handler, since canvas gestures never fire the native `input` event. A `#plot-fallback-msg` shows if Konva fails to load (`KONVA_AVAILABLE`).

- **Parcel → grid projection** — a county GeoJSON parcel polygon becomes pixel space via: `geoToLocalMeters` (lng/lat → local meters) → `rotatePoints` (by `-bearing` to match the rotated map view) → flip Y (geographic north-up → screen rows-down) → `computeBBox`. **The grid is fixed-scale (1 tile = 1 ft), not fit-to-viewport**: `buildParcelGrid` sizes a *square* whose side is the parcel's longest dimension + ~10% buffer (`GRID_MARGIN`, at least `GRID_MIN_PAD` tiles, clamped to `MAX_GRID_DIM`), centers the parcel in it, and returns `{cols, rows, polygonPx, metersPerCell}` — **no per-cell `mask` anymore** (the footprint is traced as a dashed `Konva.Line`, not cell-clipped). `metersPerCell` (= `FOOT_IN_METERS`) feeds `scaleFeetPerPixel = (metersPerCell * 3.28084) / CELL_SIZE`, the real-world scale the Measure tool uses. **`gridCols`/`gridRows` are mutable tile counts** (= feet) recomputed per parcel; `rebuildGridForParcel()` rebuilds the grid canvases + `bgLayer` and clears `cellState`, `drawLayer`, *and* the `plotConfirmed` flag — **but re-entering the Draw step with the same parcel/bearing just re-fits the view without wiping** (guarded by `lastPlotAPN`/`lastPlotBearing`; a `confirm()` fires only when the parcel/orientation actually changed under an already-drawn plan). The aerial backdrop is registered to the outline by a least-squares similarity fit (`fitSimilarity`) over corresponding parcel vertices, then shifted by a baked ground-vector ortho correction (`AERIAL_NUDGE_FT`, rotated by bearing into stage space) — the residual ~few-foot orthophoto offset is a verified *non*-datum issue (a NAD83↔WGS84 transform moves the parcels 0.00 ft), so it's tuned by eye here rather than reprojected away.

- **Map wizard** — the Site/Plot Plan section (`#siteplan`) is a 4-step flow driven by `showStep(n)`: **1 Method → 2 Select → 3 Orient → 4 Draw**. Step 1 is a `planMode` choice (`"build"` vs `"upload"`): "Build a plan" runs `startBuilder()` (which reads `#property-address` from the standalone Applicant Information card and kicks off geocoding) and reveals the Select/Orient/Draw dots; "Upload an existing plan" reveals a file input (`#plot-upload`) and hides those dots. **Mid-wizard bail-out:** every builder step (2–4) and the Konva-failure notice carries a `[data-switch-upload]` button; one shared handler switches to the upload path, returns to step 1 with the upload panel scrolled into view, and deliberately preserves the parcel selection and drawn state in case the user switches back. Map data comes from MapLibre GL + OpenFreeMap tiles + **Riverside County ArcGIS REST** services — but **parcel data is local-first**: `assets/parcels.json` (a baked extract of every parcel polygon + situs address in the service-area bbox, from county layer `50` PARCELS_CREST — the same parcel fabric as layer `40` with the assessor address columns pre-joined; regenerate via `tools/fetch-parcels.mjs`, which also computes the extract's `center`) is loaded once at module init (`parcelData`/`parcelDataReady`, same pattern as streets.json), and `locateByAddress` (address→APN geocode), `loadParcels` (the surrounding-parcels box, incl. its address labels), and `fetchParcelGeometryByAPN` all resolve from it in-memory — `bakedParcelFeature()` rebuilds records into the exact Feature shape the live queries return, so everything downstream is source-agnostic. The live endpoints (assessor table layer `50` for address→APN, parcel layer `40` for geometry) remain the **fallback** for addresses outside the baked bbox or if the extract fails to load; a `loadParcels` envelope that pokes past the baked edge goes live too, so the map never shows a truncated neighborhood. The map's starting view is the **community's center of mass** (mean of all baked parcel centroids — `parcels.json`'s `center`, with the `HOA_CENTER` constant as the pre-load fallback; keep it in sync with the value the bake script prints). **County-GIS failure story (fallback path only):** every ArcGIS `fetch` goes through `gisFetch()` (10s `AbortController` timeout so a dead endpoint can't hang the wizard), and hard failures surface `#gis-fallback-msg` — a step-2 notice carrying its own `[data-switch-upload]` route to the upload path (hidden again when a fresh lookup starts or a retry succeeds); `startBuilder()` also bails to the upload path with a message if `maplibregl` itself never loaded, mirroring the `KONVA_AVAILABLE` fallback. Note: the **single `#map-container` DOM node is physically moved** between the step-2 and step-3 containers via `target.prepend(mapContainer)` — there is only one map instance, reparented as you navigate. Step 2 locks north-up; step 3 enables rotation. **Backing up preserves state:** leaving Select for Orient captures the selection camera in `step2Camera` (invalidated by a fresh `locateByAddress`), and re-entering step 2 eases back to it north-up; entering step 3 frames the parcel ~25% closer than a plain bounds fit (`cameraForBounds` + `ORIENT_ZOOM_IN`, kept in `orientCamera`) via **one** `easeTo({center, zoom, bearing: parcelBearing})` — the bearing **must** ride in that same camera call (`fitBounds`/`easeTo` animate to bearing 0 by default, which silently un-did the dialed rotation a moment after entering Orient). Step 3 has **wheel zoom** (±50% scale around `orientCamera`, `jumpTo` carrying center+bearing so a mid-animation wheel can't strand the camera); scroll-zoom stays off globally. **The Draw step's aerial backdrop is a direct county `exportImage` fetch** (`fetchAerialBackdrop()`, kicked off on the 3→4 transition and from `restoreParcelFromDraft`): its ground bbox is computed from the same math that sizes the plot grid, so coverage is guaranteed regardless of the Orient camera — it replaced the old "screenshot the map canvas on 3→4" handoff, which clipped whenever the viewport showed less ground than the grid square (and could never restore a backdrop after reload). Keyed by APN@bearing (`lastAerialKey`), superseded in-flight by `aerialFetchSeq`, with loading overlays `#map-loading` (satellite tiles, driven by `sourcedataloading`/`sourcedata`/`idle`) and `#plot-aerial-loading` (the export fetch) — both `pointer-events:none` with a CSS appear-delay. Steps 2–4 each carry a dark `.hotkey-bar` mouse-glyph strip (steps 2/3 below the map; step 4's merged into the canvas status strip — see "Draw-step chrome"). The two map steps share one `#map-container` size (full width, `clamp(340px, 62vh, 560px)`), so the reparent doesn't jump sizes. **Orientation is drag-to-rotate directly on the map** (pointer events — mouse, touch, or pen — with pointer capture and a `pointercancel` end, horizontal drag at `ROTATE_DEG_PER_PX` = 0.5°/px, active in step 3 only; step 3 also forces `touch-action:none` on the map canvas so the browser can't hijack a touch drag, undone on step 2 where MapLibre's own pan/pinch need it back, **and makes `#map-container` a focus stop** — `tabindex=0` + `role=application`, arrow keys rotate 1°, Shift+arrow 15°, via `nudgeRotation`, announced by the `aria-live` readout) plus **±0.1° fine-nudge buttons** (`#rotate-fine-minus`/`-plus` → `nudgeRotation()`, snapping to the nearest 0.1°) with a live tenths-of-a-degree readout (`#map-rotate-fine`); the old slider / **±15°** / Reset / Auto-align panel is commented out in both `index.html` and `app.js`. **Snap-to-lot-lines**: holding **Shift** during the rotate drag (momentary) or ticking the **Snap** checkbox in the fine-nudge pill (`#rotate-snap`, sticky; ticking it also snaps immediately if in reach) sticks the bearing whenever a parcel edge comes within `SNAP_TOLERANCE_DEG` (4°) of running square to the screen grid — candidates from geometry's `computeSnapAngles` (cached per boundary ring in `parcelSnapAngles()`), applied by `applyBearingSnap()`; the readout turns green (`.rotate-fine.is-aligned`) whenever the bearing sits exactly on a candidate. The ±0.1° nudges and arrow keys deliberately don't snap (a 1° step inside the 4° zone would be trapped). Both the nudge buttons and drag live *inside* `#map-container` so they ride along when the node is reparented, and the fine control is revealed only by the `#map-step3 .rotate-fine` rule (same pattern as the orient overlay). Rotation funnels through a single `setRotation(deg)` function (normalizes mod 360 **without rounding, so fractional/tenths degrees survive**, updates `parcelBearing`, `syncRotationReadout`, and `mapInstance.setBearing`) — the drag handler, the nudge buttons, `restoreDraft()`, and `computeAutoAlignBearing()` (which squares the view to the parcel's longest boundary edge **and then picks the front-yard-down flip of the two 180°-apart candidates** using `selectedParcelRoadDir` — a unit vector toward the nearest street; falls back to the legacy `[0,180)` tie-break when that's unavailable; **Auto-align** no-ops without a `selectedParcelGeoJSON`) all funnel through it. **Nearest-street lookup is local-first:** `localNearestRoadDir()` reads a pre-baked OSM extract (`assets/streets.json`, loaded once into `streetData` via `streetDataReady`) and computes the direction in-memory with **zero network latency** for any parcel inside its bbox; only addresses *outside* that box fall through to `fetchNearestRoadDir()` → **Overpass (live OSM) over plain REST** (multi-mirror with a 7s per-endpoint `AbortController` timeout in `overpassQuery`). The lookup is kicked off in the parcel-click handler the moment the boundary resolves (Step 2) and cached in `selectedParcelRoadDir`; the Step 2→3 auto-align and the Auto-align button both `roadDirReady.then(...)`-wait for it (the 2→3 path caps the wait at 2.5s, and re-applies a late-arriving correction unless `userAdjustedRotation`). Why OSM and not the obvious sources: the county centerline layer (`OpenData/Transportation/0`) is **missing Fairway Canyon's residential streets entirely** (its "AARON" segments are 16+ km away — a different road), and the basemap's own OpenMapTiles `transportation` vector tiles come back empty from `querySourceFeatures` at align time (tile load/visibility timing), so neither is reliable here. **`assets/streets.json`** is a one-time Overpass extract (`way[highway=…]` within bbox `[-117.085,33.925,-117.01,33.985]`, ~901 streets, ~300 KB, © OpenStreetMap contributors/ODbL); regenerate it with a bounded Overpass `out geom;` query if the service area changes. Note the loader `fetch()`es it same-origin, so it only loads over http(s) (`python -m http.server`), **not** `file://` — under `file://` it silently falls back to live Overpass. Step 3 also dismisses the "Selected parcel" popup (`selectedPopup`, created on parcel click) on entry, and shows a transparent **Back Yard / Front Yard overlay** (`.map-orient-overlay`) — that overlay lives *inside* `#map-container` so it rides along when the node is reparented, but is `display:none` by default and only revealed by the `#map-step3 .map-orient-overlay` rule. The map's zoom/compass control buttons are likewise hidden in step 3 only via `#map-step3 .maplibregl-ctrl-top-right { display:none }` (kept for step 2's parcel selection). Gotcha: builder-only step dots are hidden via inline `style.display` (not the `hidden` attribute), because `.plot-steps-nav__dot` sets `display:flex`, which would override `[hidden]`.

- **Keyboard access (Sprint 5 review outcome)** — two builder interactions are inherently pointer-only and are documented as such, not worked around: **parcel selection** (clicking a lot on the MapLibre canvas in step 2) and **freehand drawing** on the Konva surface. The complete keyboard-only route is the **upload path**: every control on it (plan-choice cards, file inputs, all form fields, the Draw/Type signature toggle and typed signature) is a native focusable element, and every builder step carries a focusable `[data-switch-upload]` bail-out. What *was* made keyboard-accessible: Orient-step rotation (arrow keys on the focused map container, see above) and the owner signature (Type method). When adding new interactions, keep the upload path's keyboard completeness intact.

## Gotchas

- **Form-content edits go in the data constants** (`ACKS`/`PALETTE`/`STAMPS`/`TOOL_MODES`/`DATES`/`PHOTO_SPECS`/`CATEGORIES`), not the DOM, which is generated from them.
- **`Konva.toJSON()` returns a *string*, not an object.** `serializePlot()` does `JSON.parse(drawLayer.toJSON()).children` before nesting it into the draft object — nesting the raw string produces double-escaped JSON bloat in `localStorage` (and in `renderPlotImage()`'s offscreen-stage rebuild).
- **Cross-module imports from `app.js` are function-only and event-time.** plot-editor.js and map-wizard.js import `scheduleAutosave`/`updateProgress`/`setFieldError`/`refreshPacketUI` back from the entry module (an intentional ESM cycle). Hoisted function declarations make this safe; importing a `const`/`let` from `app.js` and touching it at another module's top level throws a TDZ ReferenceError at page load. Conversely, the wizard's exported state (`planMode`, `parcelBearing`, `selectedAPN`, …) is read cross-module as live bindings — only map-wizard.js itself may assign them (use its setters: `setPlanMode`, `setRotation`, `restoreParcelFromDraft`).
- **A drag/click gesture that *releases* past the canvas edge never fires the Konva stage's own `pointerup`** — native pointer events only reach elements the pointer is still over, so without a fallback, an in-progress shape/callout/measurement is silently abandoned instead of committed. `initPlotStage()` also binds `onStagePointerUp` on `window` as a fallback; the handler is idempotent (no-ops once state is already clear) so double-binding is safe.
- **Materials are raster, annotations are vector — two separate persistence + undo paths.** The material grid lives in the `cellState` `Map` painted onto `paintCanvas` (so Fill is a *real* 4-way `floodFill`, not the old click-a-shape-to-recolor trick); callouts/measurements live as Konva nodes on `drawLayer`. `collect()`, `snapshotState()`/`applyHistorySnapshot()`, and `renderPlotImage()` each have to handle **both** — forget one and either the painted grid or the annotations silently drops out of the draft / undo / print.
- **`[hidden]` vs author `display`:** hiding an element with the `hidden` attribute only works if no author rule sets `display` on it. `.layout`/`.modal` (`display:grid`), the status strip's `.plot-status__brush`/`.plot-status__stamps` (flex/grid), `.palette-pop` (flex), the attached-photo `.photo-thumb` (flex), and the multi-shot `.photo-thumb-list` (grid) all need explicit `[hidden] { display:none }` rules, and builder-only step dots use inline `style.display` instead of `hidden` (`.plot-steps-nav__dot` is `display:flex`). Same gotcha, eight places.
- `ISSUES.md` is the running work log; **#3** (good-vs-bad photography examples) is now wired into the Photos section as **placeholder frames** — the structure exists, the real comparison images are still pending.
- **MapLibre only runs one camera animation at a time.** The step 2→3 transition in `showStep()` frames the parcel via one `easeTo({center, zoom, bearing})` ~100ms after the step switches. Auto-align used to call a separate `mapInstance.easeTo({bearing})` for the rotation — the framing animation starting shortly after cancelled it mid-flight and froze the bearing whatever it had reached at that instant (usually still ~0°). Fix: the target bearing is folded into the *same* camera call instead of a standalone animation (the Orient wheel-zoom handler likewise uses a single `jumpTo` carrying center+zoom+bearing). If you ever add another camera-animating call near a `showStep()` transition, check whether it collides with an in-flight one the same way.
- **Never trust `e.features[0].geometry` from a MapLibre GeoJSON-source click handler for anything precision-sensitive.** GeoJSON sources are internally re-tiled (geojson-vt) for rendering, and the feature geometry returned by a click/`queryRenderedFeatures` is that simplified render geometry, not the original. The parcel-click handler resolves `selectedParcelGeoJSON` from `parcelsData` (the raw fetched GeoJSON) instead, re-fetching by APN via `fetchParcelGeometryByAPN()` if it's missing there — this was the root cause of the Draw-step grid once rendering visibly narrower than the real parcel.
