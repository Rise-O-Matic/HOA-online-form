# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A **static, client-side mockup** of the Fairway Canyon HOA Architectural Review Committee (ARC) application, rebuilt from a printed PDF (`docs/source-form.pdf`) as a web form for GitHub Pages. **There is no backend.** Nothing is transmitted: drafts persist to `localStorage`, "submitting" generates a printable preview or opens a `mailto:`, and data can be exported as JSON.

Plain HTML/CSS/JS ‚Äî **no build step, no npm dependencies, no linter.** The behavior is native **ES modules** under `assets/` (loaded directly by the browser via `<script type="module">`); the pure geometry module has a **`node --test` unit suite** in `tests/`, and the repo-root `package.json` exists *only* to set `"type": "module"` for Node tooling (no deps, no scripts beyond `test`). MapLibre GL JS (maps) and Konva.js (the plot-plan drawing stage ‚Äî a raster material grid plus vector annotations) are **vendored as exact pinned builds in `assets/vendor/`** (version in the filename; to upgrade, download the new dist file there, update the `index.html` tags, and commit both together ‚Äî never reintroduce a floating CDN tag). The one remaining CDN dependency is Google Fonts, which degrades gracefully to system fonts. (There is one dev-time Node script, `tools/fetch-streets.mjs`, but it is not part of the app or its build ‚Äî see Commands.)

## Commands

- **Run locally:** `python -m http.server 8000`, then visit `http://localhost:8000`. Use a server rather than opening `index.html` via `file://` ‚Äî ES modules won't load at all under `file://` (CORS), and the county GIS `fetch()` calls and map tiles need an http(s) origin anyway.
- **Deploy:** GitHub Pages serves the repo root (`.nojekyll` disables Jekyll processing). The live branch is **`master`**.
- **Tests:** `node --test` (or `npm test`) runs `tests/geometry.test.js` ‚Äî 29 cases over the pure geometry (grid sizing/clamps, similarity fit, auto-align, flood-fill walls/coverage rules). Run it after touching `assets/geometry.js`, and extend it when moving more pure logic there.
- **Sanity check:** always run `node --check` on every `assets/*.js` file you edited ‚Äî it catches syntax errors before you hand a change to the user (who does the real browser verification). No build step: edit the source files and reload.
- **Regenerate the baked street extract:** `node tools/fetch-streets.mjs` rewrites `assets/streets.json` from live Overpass (it tries several public mirrors in order). Widen the `BBOX` in that script if the service area grows. Dev-time tool only ‚Äî the deployed app just `fetch()`es the committed JSON.

### Verification / testing this agent CAN'T do well

**Ask the user to test map-wizard and drawing-surface changes for you ‚Äî don't burn turns trying to drive them via browser automation.** The map wizard (Select ‚Üí Orient ‚Üí Draw) and the Konva plot surface are effectively un-testable from Claude Code's browser automation in this environment, for concrete reasons learned the hard way:
- The MapLibre map frequently sits **below the fold and won't scroll into the clickable viewport**, so screen-clicking a parcel is unreliable.
- The dev server (`python -m http.server`) sends no cache headers, so the browser **serves a stale `assets/app.js` even after edits + reload** ‚Äî you have to cache-bust the `<script src>` to load fresh code, which is throwaway churn.
- Driving the flow programmatically needs a **temporary `window.__dbg` hook** (module internals aren't otherwise reachable), plus the county GIS lookup, the auto-align camera animation, and the aerial snapshot capture all have real latency/timing that browser automation races against.
- Screenshots intermittently time out or render faded/blank in this sandbox.

So: implement the change, `node --check` the edited modules (+ `node --test` if geometry changed), reason carefully about the geometry/timing, then **hand it to the user to verify in their own browser** (they have the real map, real parcels, and can eyeball scale/alignment instantly). Reserve heavy browser-automation verification for the static form sections that actually sit in the viewport ‚Äî one workaround for the stale-cache problem there: serve on a **fresh port** for each verification round.

## Architecture

Markup is `index.html`; presentation is `assets/styles.css` (~1200 lines, organized in clearly-labeled `/* === SECTION === */` blocks ending with a `@media print` block that styles the print output). Behavior is **five ES modules under `assets/`**, loaded natively (entry: `<script type="module" src="assets/app.js">`; the vendored Konva/MapLibre stay classic scripts that define globals):

- **`utils.js`** ‚Äî `$`, `$$`, `esc`. Imported everywhere.
- **`geometry.js`** ‚Äî the pure, DOM-free math: scale constants (`CELL_SIZE`, `FOOT_IN_METERS`, `GRID_*`, `MAX_GRID_DIM`), parcel projection (`geoToLocalMeters`/`rotatePoints`/`computeBBox`/`buildParcelGrid`), `fitSimilarity`, `computeAutoAlignBearing(ring, roadDir)` (road direction is a **parameter**, not module state), `closestPointOnSegment`, and the flood-fill core (`segmentsIntersect`, `connectorBlocked`, `pointOnSegment`/`pointOnAnyWall`, `computeFloodFill`). **Unit-tested** (`tests/geometry.test.js`); keep it dependency-free and put new pure logic here.
- **`plot-editor.js`** ‚Äî the Konva drawing surface (everything in "Plot drawing surface" below) plus its data constants `PALETTE`/`ICON`/`TOOL_MODES`. Owns the grid/canvas/undo state. Cross-module API: `rebuildGridForParcel(feature, bearing, apn)`, `setPlotBackdrop(dataUrl, parcelPx)` (the wizard's aerial handoff), `plotUsed()`, `renderPlotImage()`, `serializePlot()`/`restorePlot(plot)` (persistence handshake).
- **`map-wizard.js`** ‚Äî the Site/Plot Plan 4-step flow, MapLibre map, county-GIS lookups, rotation, plan-mode choice, OSM street lookup. Exports live-binding state the form reads (`planMode`, `parcelBearing`, `selectedAPN`, `selectedParcelGeoJSON`, `mapInstance`, `plotUploadInput`) plus `setPlanMode`, `setRotation`, `restoreParcelFromDraft(feature, apn, bearing)`.
- **`app.js`** ‚Äî entry + form/persistence: `ACKS`/`DATES`/`PHOTO_SPECS` data, `SignaturePad`, neighbors, photos, progress meter, packet status, `collect()`, drafts, validation, preview/print/email/JSON outputs, landing, init. Exports `scheduleAutosave`, `updateProgress`, `setFieldError`, `refreshPacketUI` for the other modules.

**Module-evaluation order is load-bearing**: `app.js` imports plot-editor before map-wizard, so top-level init runs plot editor ‚Üí wizard ‚Üí form, exactly the old single-file order. plot-editor and map-wizard import those four `app.js` functions back ‚Äî a **deliberate ESM cycle** that is safe only because they're hoisted function declarations called at event time; never *read a non-function binding* from `app.js` at another module's top level (TDZ crash at load).

Content is data-driven from constants ‚Äî **edit these arrays to change form content**, not the generated DOM:
- `ACKS` (app.js) ‚Äî the 8 acknowledgment statements (HTML strings, rendered into `#acks`)
- `PALETTE` (plot-editor.js) ‚Äî plot-plan materials (`id`/`label`/`color`), rendered into the material toolbar (`#palette`); `TOOL_MODES` is the separate list of drawing-tool modes (paint ¬∑ rect ¬∑ erase ¬∑ fill ¬∑ line ¬∑ callout ¬∑ measure ¬∑ select ¬∑ pan, each paired with an inline SVG from the `ICON` map **and a one-line `hint` HTML string** ‚Äî `updateToolHint()` swaps it into `#tool-hint` above the canvas whenever `setActiveMode()` runs), rendered into `#tool-palette`
- `DATES` (app.js) ‚Äî the 2026 review-date table (rendered into `#dates-body`, which lives on the **landing page**)
- `PHOTO_SPECS` / `PHOTO_MATERIAL` (app.js) ‚Äî the per-area photo requests for the Photos section (each shot has `id`/`title`/`instr`/`good`/`bad`); editing these changes which photos are requested and their instructions

The form is a **multi-section single page**, gated behind a **landing/cover screen**. DOM/section order (and card numbers): landing ‚Üí `01` Applicant ¬∑ `02` Site/Plot Plan ¬∑ `03` Proposed Change ¬∑ `04` Adjacent Owners ¬∑ `05` Photos ¬∑ `06` Required Submissions (live packet status + confirm) ¬∑ `07` Acknowledgments ¬∑ `08` Review & Submit (`#packet` ‚Äî packet checklist + guided finish; replaced the old sticky three-button submit bar). The sidenav, card numbers, and DOM order must agree when sections are added/reordered.

### Key subsystems (landing/photos/persistence/packet/validation in app.js; plot surface in plot-editor.js; wizard/geocoding in map-wizard.js)

- **Landing / form reveal** ‚Äî a full-screen cover (`#landing`, between the masthead and the form) holds the unpacked required-submissions overview + the 2026 review-dates table + how-to-submit/assistance. The whole form (`.layout`, `id="form-layout"`) starts with the `hidden` attribute; `enterForm()` (wired to `#start-application`) unhides it, and `showLanding()` (wired to the sidenav `#view-landing` link) returns to it. **Returning users skip the landing:** `restoreDraft()` returns a boolean and `enterForm()` is called when a draft was restored.

- **Photos** (`#photos-section`) ‚Äî a **questionnaire drives which photos are requested**. The two questions (which areas: `area_front/back/side/exterior` checkboxes; plus a `photoMaterial` yes/no radio) toggle pre-built per-area groups via `refreshPhotoGroups()` (it flips each `.photo-group`'s `hidden`, and shows/hides the `#photo-empty` placeholder). Groups/blocks are generated once by `buildPhotoRequests()` from `PHOTO_SPECS`/`PHOTO_MATERIAL`, so already-attached files survive a questionnaire change (groups are hidden, not destroyed). Each shot has its own `<input type=file data-photo-input=ID>` + a ‚úì/‚úó example pair (**placeholder frames** ‚Äî real comparison images pending, ISSUES #3). `collect()` emits `photoAreas`, `photoMaterial`, and a `photos` map (`id ‚Üí filename`); `files` is the flat list of attached photo names for preview/print. As with all file inputs, restore can only show the prior filename (browsers can't repopulate a file input), shown as an `is-prior` status.

- **`SignaturePad`** (the one class) ‚Äî DPR-aware `<canvas>` signature capture via pointer events. Serializes to a PNG dataURL (`toDataURL`/`fromDataURL`) so signatures survive localStorage round-trips. Instances are tracked in the `sigPads` map, keyed by the element's `data-sigpad` attribute. **Hidden-safe sizing:** `resize()` no-ops while the pad is `display:none` (the landing gate, or the Type method below) so a 0√ó0 rect can't clobber the bitmap; `enterForm()` sizes the pads on reveal, and `fromDataURL` on a hidden pad parks the url in `this.pending` (applied on the next resize, and passed through by `toDataURL()` so a hidden-pad draft round-trip doesn't lose ink). **Signature method toggle (Sprint 5):** the owner signature offers Draw/Type (`#sig-method-draw`/`#sig-method-type` ‚Üí `setSigMethod()`, state in `sigMethod`); Type reveals `#typed-signature`, whose typed full legal name is the keyboard/AT-accessible e-signature alternative. `ownerSignatureProvided()` is the method-aware "signed?" check used by validation and the progress meter; both inputs persist in the draft (`ownerSigMethod`/`ownerTypedSignature`, additive to `.v4`), and preview/print render whichever method is active (typed names get a script face, plus a "signed electronically by typing name" label in print).

- **Plot drawing surface (raster grid painter + Konva annotations)** ‚Äî the "Draw Your Plan" step (`#plot-konva-host`) is a `Konva.Stage` with **four** layers, bottom-to-top: `bgLayer` (`listening:false`; aerial snapshot + dashed parcel outline), `gridLayer` (`listening:false`; two stacked `Konva.Image`s wrapping offscreen `<canvas>`es ‚Äî `gridLinesCanvas` graph paper + `paintCanvas` painted materials ‚Äî with `mixBlendMode:"multiply"` set on the layer's own DOM canvas so materials read *through* the aerial on screen, while `toDataURL()` ignores the blend so print stays opaque), `drawLayer` (Konva callout + measurement groups + Line-tool lines **only**), and `overlayLayer` (in-progress ghost shapes + the Select-highlight rect ‚Äî excluded from persistence/export just by living on its own layer). **Scale is fixed: 1 grid tile = 1 ft = `CELL_SIZE` (8) px at 100% zoom.** A bigger lot yields a *bigger grid you pan/zoom around*, never a shrink-to-fit one, so a foot is always the same size on screen. **Materials are raster**, held in the `cellState` `Map` (`"c,r" ‚Üí materialId`) and drawn with real cell ops dispatched from `onStagePointerDown/Move/Up`: `paintBrush`/`paintStroke` (a square brush of `brushSize` ft, interpolated so a fast drag leaves no gaps), `fillRect` (Rectangle rubber-bands a block; **Shift** on Paint draws a straight line), and a genuine 4-way **`floodFill`** paint-bucket. Right-click / Ctrl(‚åò)-click is a quick eraser from any tool. **Annotations are vector Konva**: callouts are a `Konva.Group` (`Label`+`Tag`+`Text` plus a free-angle `Konva.Arrow` leader, since `Tag.pointerDirection` only supports 4 cardinals) measurements a `Konva.Group` (`Konva.Arrow` + a `Konva.Text` distance label via `scaleFeetPerPixel`), and the **Line** tool a plain `Konva.Line` (`kind:"line"`) whose endpoints `snapToGrid` to the nearest grid-line intersection and whose stroke is a real-world 2 in (`gridLineWidth()` = `LINE_WIDTH_FEET / scaleFeetPerPixel`, so it stays 2 in at any zoom) with a fat `hitStrokeWidth` so the hairline is still clickable ‚Äî and Lines also act as **flood-fill walls at any angle** (triangles included): `floodFill` floods at `FILL_SUBSAMPLE`¬≤ (4√ó4) sub-tile resolution and blocks a step whenever the connector between two subcell centres crosses a Line (`collectLineSegments`/`segmentsIntersect`/`connectorBlocked`) ‚Äî watertight for straight segments since a straight connector crosses a straight segment iff its endpoints are on opposite sides, so a closed shape whose corners share snapped vertices fully contains the fill ‚Äî then paints a whole tile iff **>50% of its subcells are inside** (the "more than half the tile is within bounds" rule, which also subsumes axis-aligned walls and 100%-interior tiles). Coverage = reached subcells **plus the on-wall strip** (`pointOnAnyWall`) that the conservative touch-block otherwise erodes ‚Äî without this a diagonal-cut tile reads <50% and leaves a visible gap, so triangles wouldn't fill to their edge. Filling the *other* side of a wall can't overwrite already-painted edge tiles (the material-region check blocks the second flood from entering them), so the **first fill keeps the shared edge ‚Äî no gap, no overwrite**. The fill is seeded from the subcell under the click so it starts on the clicked side of a wall; `attachShapeInteractions` wires Select (drag-to-move) and Erase (click-to-delete) per node, and annotations stay `listening(false)` outside those two modes so a paint stroke passes straight through to the grid beneath. **Pan/zoom is DOM-scroll based**: `applyStageSize` scales the stage by `viewZoom` and the `#plot-konva-host` element scrolls; `minZoom`/`maxZoom`/`clampZoom` bound you between "no whitespace" and a `SAFE_CANVAS_PX` memory cap; `fitView` ("Fit") zooms out and centers on the parcel; the Pan tool / middle-mouse drag adjusts native scroll. **Touch: a two-finger pinch zooms/pans** while one finger keeps using the active tool ‚Äî host-level capture-phase pointer tracking (`touchPts`/`pinch`) sets the pinch flag before Konva's own `pointerdown` could start a draw with the second finger; `beginPinch()` cancels any in-progress gesture and rolls a half-painted stroke back through the undo point recorded at its start; `onStagePointerDown/Move` early-return while a pinch is live; zoom+pan is one formula (pin the content point under the fingers' starting midpoint to the current midpoint). **Undo/redo (`snapshotState`) captures BOTH** `cellState` (as pairs) **and** `drawLayer.toObject().children` onto a capped (50-deep) stack, recorded via `recordUndoPoint()` at the *start* of each gesture, not the end. `renderPlotImage()` (used by `buildPreview()`/`buildPrintHTML()`) composites a **throwaway offscreen `Konva.Stage`** ‚Äî never the live one ‚Äî from `gridLinesCanvas` + `paintCanvas` + the parcel outline + `drawLayer.toJSON()`, so pan/zoom state and overlay ghosts can never leak into print. Autosave (`scheduleAutosave()`) is called explicitly from every tool's commit handler, since canvas gestures never fire the native `input` event. A `#plot-fallback-msg` shows if Konva fails to load (`KONVA_AVAILABLE`).

- **Parcel ‚Üí grid projection** ‚Äî a county GeoJSON parcel polygon becomes pixel space via: `geoToLocalMeters` (lng/lat ‚Üí local meters) ‚Üí `rotatePoints` (by `-bearing` to match the rotated map view) ‚Üí flip Y (geographic north-up ‚Üí screen rows-down) ‚Üí `computeBBox`. **The grid is fixed-scale (1 tile = 1 ft), not fit-to-viewport**: `buildParcelGrid` sizes a *square* whose side is the parcel's longest dimension + ~10% buffer (`GRID_MARGIN`, at least `GRID_MIN_PAD` tiles, clamped to `MAX_GRID_DIM`), centers the parcel in it, and returns `{cols, rows, polygonPx, metersPerCell}` ‚Äî **no per-cell `mask` anymore** (the footprint is traced as a dashed `Konva.Line`, not cell-clipped). `metersPerCell` (= `FOOT_IN_METERS`) feeds `scaleFeetPerPixel = (metersPerCell * 3.28084) / CELL_SIZE`, the real-world scale the Measure tool uses. **`gridCols`/`gridRows` are mutable tile counts** (= feet) recomputed per parcel; `rebuildGridForParcel()` rebuilds the grid canvases + `bgLayer` and clears both `cellState` and `drawLayer` ‚Äî **but re-entering the Draw step with the same parcel/bearing just re-fits the view without wiping** (guarded by `lastPlotAPN`/`lastPlotBearing`; a `confirm()` fires only when the parcel/orientation actually changed under an already-drawn plan). The aerial backdrop is registered to the outline by a least-squares similarity fit (`fitSimilarity`) over corresponding parcel vertices, then shifted by a baked ground-vector ortho correction (`AERIAL_NUDGE_FT`, rotated by bearing into stage space) ‚Äî the residual ~few-foot orthophoto offset is a verified *non*-datum issue (a NAD83‚ÜîWGS84 transform moves the parcels 0.00 ft), so it's tuned by eye here rather than reprojected away.

- **Map wizard** ‚Äî the Site/Plot Plan section (`#siteplan`) is a 4-step flow driven by `showStep(n)`: **1 Method ‚Üí 2 Select ‚Üí 3 Orient ‚Üí 4 Draw**. Step 1 is a `planMode` choice (`"build"` vs `"upload"`): "Build a plan" runs `startBuilder()` (which reads `#property-address` from the standalone Applicant Information card and kicks off geocoding) and reveals the Select/Orient/Draw dots; "Upload an existing plan" reveals a file input (`#plot-upload`) and hides those dots. **Mid-wizard bail-out:** every builder step (2‚Äì4) and the Konva-failure notice carries a `[data-switch-upload]` button; one shared handler switches to the upload path, returns to step 1 with the upload panel scrolled into view, and deliberately preserves the parcel selection and drawn state in case the user switches back. Map data comes from MapLibre GL + OpenFreeMap tiles + **Riverside County ArcGIS REST** services: the assessor table (layer `50`) resolves address ‚Üí APN, and the parcel layer (`40`) returns polygon geometry. **County-GIS failure story:** every ArcGIS `fetch` goes through `gisFetch()` (10s `AbortController` timeout so a dead endpoint can't hang the wizard), and hard failures surface `#gis-fallback-msg` ‚Äî a step-2 notice carrying its own `[data-switch-upload]` route to the upload path (hidden again when a fresh lookup starts or a retry succeeds); `startBuilder()` also bails to the upload path with a message if `maplibregl` itself never loaded, mirroring the `KONVA_AVAILABLE` fallback. Note: the **single `#map-container` DOM node is physically moved** between the step-2 and step-3 containers via `target.prepend(mapContainer)` ‚Äî there is only one map instance, reparented as you navigate. Step 2 locks north-up; step 3 enables rotation. **Backing up preserves state:** leaving Select for Orient captures the selection camera in `step2Camera` (invalidated by a fresh `locateByAddress`), and re-entering step 2 eases back to it north-up; re-entering step 3 re-frames the parcel via `fitBounds({..., bearing: parcelBearing})` ‚Äî the bearing **must** stay in that call (`fitBounds` animates to bearing 0 by default, which silently un-did the dialed rotation a moment after entering Orient). The two steps share one `#map-container` size (full width, `clamp(340px, 62vh, 560px)`), so the reparent doesn't jump sizes. **Orientation is drag-to-rotate directly on the map** (pointer events ‚Äî mouse, touch, or pen ‚Äî with pointer capture and a `pointercancel` end, horizontal drag at `ROTATE_DEG_PER_PX` = 0.5¬∞/px, active in step 3 only; step 3 also forces `touch-action:none` on the map canvas so the browser can't hijack a touch drag, undone on step 2 where MapLibre's own pan/pinch need it back, **and makes `#map-container` a focus stop** ‚Äî `tabindex=0` + `role=application`, arrow keys rotate 1¬∞, Shift+arrow 15¬∞, via `nudgeRotation`, announced by the `aria-live` readout) plus **¬±0.1¬∞ fine-nudge buttons** (`#rotate-fine-minus`/`-plus` ‚Üí `nudgeRotation()`, snapping to the nearest 0.1¬∞) with a live tenths-of-a-degree readout (`#map-rotate-fine`); the old slider / **¬±15¬∞** / Reset / Auto-align panel is commented out in both `index.html` and `app.js`. Both the nudge buttons and drag live *inside* `#map-container` so they ride along when the node is reparented, and the fine control is revealed only by the `#map-step3 .rotate-fine` rule (same pattern as the orient overlay). Rotation funnels through a single `setRotation(deg)` function (normalizes mod 360 **without rounding, so fractional/tenths degrees survive**, updates `parcelBearing`, `syncRotationReadout`, and `mapInstance.setBearing`) ‚Äî the drag handler, the nudge buttons, `restoreDraft()`, and `computeAutoAlignBearing()` (which squares the view to the parcel's longest boundary edge **and then picks the front-yard-down flip of the two 180¬∞-apart candidates** using `selectedParcelRoadDir` ‚Äî a unit vector toward the nearest street; falls back to the legacy `[0,180)` tie-break when that's unavailable; **Auto-align** no-ops without a `selectedParcelGeoJSON`) all funnel through it. **Nearest-street lookup is local-first:** `localNearestRoadDir()` reads a pre-baked OSM extract (`assets/streets.json`, loaded once into `streetData` via `streetDataReady`) and computes the direction in-memory with **zero network latency** for any parcel inside its bbox; only addresses *outside* that box fall through to `fetchNearestRoadDir()` ‚Üí **Overpass (live OSM) over plain REST** (multi-mirror with a 7s per-endpoint `AbortController` timeout in `overpassQuery`). The lookup is kicked off in the parcel-click handler the moment the boundary resolves (Step 2) and cached in `selectedParcelRoadDir`; the Step 2‚Üí3 auto-align and the Auto-align button both `roadDirReady.then(...)`-wait for it (the 2‚Üí3 path caps the wait at 2.5s, and re-applies a late-arriving correction unless `userAdjustedRotation`). Why OSM and not the obvious sources: the county centerline layer (`OpenData/Transportation/0`) is **missing Fairway Canyon's residential streets entirely** (its "AARON" segments are 16+ km away ‚Äî a different road), and the basemap's own OpenMapTiles `transportation` vector tiles come back empty from `querySourceFeatures` at align time (tile load/visibility timing), so neither is reliable here. **`assets/streets.json`** is a one-time Overpass extract (`way[highway=‚Ä¶]` within bbox `[-117.085,33.925,-117.01,33.985]`, ~901 streets, ~300 KB, ¬© OpenStreetMap contributors/ODbL); regenerate it with a bounded Overpass `out geom;` query if the service area changes. Note the loader `fetch()`es it same-origin, so it only loads over http(s) (`python -m http.server`), **not** `file://` ‚Äî under `file://` it silently falls back to live Overpass. Step 3 also dismisses the "Selected parcel" popup (`selectedPopup`, created on parcel click) on entry, and shows a transparent **Back Yard / Front Yard overlay** (`.map-orient-overlay`) ‚Äî that overlay lives *inside* `#map-container` so it rides along when the node is reparented, but is `display:none` by default and only revealed by the `#map-step3 .map-orient-overlay` rule. The map's zoom/compass control buttons are likewise hidden in step 3 only via `#map-step3 .maplibregl-ctrl-top-right { display:none }` (kept for step 2's parcel selection). Gotcha: builder-only step dots are hidden via inline `style.display` (not the `hidden` attribute), because `.plot-steps-nav__dot` sets `display:flex`, which would override `[hidden]`.

- **Address geocoding** ‚Äî `parseAddress()` regex-splits "number + street name" and queries the county assessor endpoint. It is **hardcoded to `CITY='BEAUMONT'`** and Riverside County URLs ‚Äî this app is not geographically portable without changing those endpoints and the city filter.

- **Persistence** ‚Äî `collect()` serializes the entire form (text fields, checkboxes, neighbors, all signatures as dataURLs, and the plot) into one object. The **plot is two things now**: `plot: {version: PLOT_VERSION, cell, cols, rows, cells: [...cellState], annotations}` ‚Äî the raster grid as `Map` pairs **plus** the vector annotations ‚Äî assembled by `serializePlot()` (plot-editor.js); persistence stamps the `version` ‚Äî and `plotMeta` with parcel coords/APN/bearing. `saveDraft`/`restoreDraft` read/write `localStorage` under **`DRAFT_KEY = "fairwayCanyonArcDraft.v4"`**; **`PLOT_VERSION = 4`** is kept in lockstep with the key's suffix (drafts written before the Sprint-3 reconcile carry `plot.version` 3 in the *identical* format, so restore accepts 3‚Äì4). Autosave is debounced 1200ms via `scheduleAutosave()`, called both from the form's native `input` listener and directly from every canvas tool's commit handler. **Autosave failure is visible:** `saveDraft()` un-hides the persistent `#save-warning` sidenav indicator when `localStorage.setItem` throws (quota/blocked ‚Äî signature PNGs + plot state can approach the ~5 MB quota) and hides it again on the next successful save. **Draft deletion goes through `deleteDraft()`**, which removes the current key *and* the legacy keys ‚Äî removing only `DRAFT_KEY` would let the one-time legacy migration resurrect a stale draft on the next load; it's called from the sidenav Clear button and the Section 08 post-submit cleanup panel (`#post-submit-cleanup`, revealed by `finishEmail()` since the draft holds PII including the signature image). Restoring re-adopts the saved parcel as the wizard's selection via `restoreParcelFromDraft(feature, apn, bearing)` ‚Äî which also repopulates `selectedParcelGeoJSON`, without which the first post-restore autosave would drop `parcelCoords` from the draft (a real bug until Sprint 4) ‚Äî rebuilds the grid (`rebuildGridForParcel`), then rehydrates the drawing via `restorePlot(plot)` (plot-editor.js): `loadCells(plot.cells)` plus `hydrateShapesInto` ‚Üí `Konva.Node.create` + `attachShapeInteractions` (serialized Konva JSON carries no listeners, so skipping the re-attach leaves shapes visible but inert). If no `.v4` draft exists, `restoreDraft()` walks **`LEGACY_DRAFT_KEYS = [".v3", ".v2"]` once**: every non-plot field migrates losslessly, but the drawing is **left empty** ‚Äî neither the old Konva-vector `.v3` plan nor the pre-Konva `.v2` cell-grid maps onto the new fixed-scale grid ‚Äî with a one-time notice (`#plot-migration-notice`), and the migrated draft is immediately re-saved under `.v4` so the migration runs once per browser.

- **Packet status (Section 06 + Section 08)** ‚Äî one set of helpers is the single source of truth for "what's in the packet": `plotProvided()` (drawn `plotUsed()` or uploaded `#plot-upload`, depending on `planMode`), `photoChecklist()` (every questionnaire-requested shot with its attached filename or `null`), `neighborFormFiles()`, `sketchesConfirmed()` (the one remaining meaningful manual checkbox), and `packetMissingList()`. They feed four consumers: the Section 06 **live status rows** (`.reqstat[data-req]`, updated by `refreshRequirementRows()` ‚Äî plot/photos/neighbors are *derived*, not checkboxes; only sketches + fee are still manual), the Section 08 **"Your packet" list** (`renderPacket()`, with per-photo subrows), the **email soft-gate**, and the **mailto attachment manifest**. `refreshPacketUI()` re-renders all of it and is called from `updateProgress()` (which itself runs on form input/change, on document pointerup for canvas gestures, and from `setPlanMode`). `collect()` derives `submissions.req_plot/req_photos/req_neighbors` from these helpers, so preview/print/JSON reflect real state. `pdfSaved` is a **session-only flag** flipped by `printPreview()` (we can't observe an actual file save, and files don't persist across reloads) ‚Äî after a refresh the gate will again list the form PDF as missing; that's intended.

- **Guided finish (Section 08)** ‚Äî Step 1 `#finish-pdf-btn` (validate ‚Üí `printPreview()`; a `type=submit` "Preview first" button opens the preview modal instead) then Step 2 `#email-btn`. **Both email buttons (`#email-btn` and the modal's `#do-email`) funnel through one `startEmailFlow()`**: validate (`validateOrFocus()`, shared with the other paths) ‚Üí `packetMissingList(true)` ‚Üí if anything's missing, open the **soft-gate** `#packet-gate-modal` (lists gaps incl. each missing photo by name; `#gate-send-anyway` is the explicit override) ‚Üí `finishEmail()` ‚Üí `openMailto()`. Don't reintroduce a second email path that skips validation or the gate. `finishEmail()` also reveals `#post-submit-cleanup` ‚Äî the offer to delete the saved draft once the packet is sent (deliberately not automatic: we can't observe the send, and the user may need the draft to revise).

- **Output paths** ‚Äî four ways data leaves the form, all client-side: the **preview modal** (`buildPreview`), a **single-page print window** (`buildPrintHTML` ‚Üí `printPreview` opens `window.open` and writes a self-contained doc with an inline `<style>` block ‚Äî print CSS lives there, not in styles.css; it also sets `pdfSaved`), a **`mailto:`** link (`openMailto` ‚Äî its body carries an **attachment manifest** enumerating every file to attach, plus a "still outstanding" list when the gate was overridden), and **JSON download** (now the sidenav "Export JSON" button, out of the primary flow). `collect()` is the single source feeding all four. There is also a fifth, separate print window: `printNeighborForm()`/`buildNeighborFormHTML()` generates the **wet-signature form for adjacent owners** (the Adjacent Owners section collects names/addresses digitally, but signatures are gathered on this printout and re-attached via `#neighbor-form-file`).

- **Validation / progress** ‚Äî `validate()` gates preview/PDF/email: it walks required fields, applies `PHONE_RE`/`EMAIL_RE` format checks, and paints per-field messages via `setFieldError` (cleared on input); `validateOrFocus()` wraps it with the shared scroll-to-first-error + status-line behavior. Packet completeness is deliberately **not** part of `validate()` ‚Äî it soft-gates the email step instead (missing items warn, never hard-block). `updateProgress()` drives the sidenav completion meter from the `[required]` census plus the owner acknowledgment signature (method-aware via `ownerSignatureProvided()` ‚Äî drawn ink or typed name; note `#typed-signature` is deliberately *not* `[required]`, since its requiredness is conditional on `sigMethod`, so both `validate()` and the error-clearing input listener special-case it) **plus three packet items** (plot provided, all questionnaire-requested photos attached, neighbor form attached) ‚Äî so 100% is unreachable with an empty packet. `#ack-date` defaults to today at init unless the restored draft carried a value.

- **Keyboard access (Sprint 5 review outcome)** ‚Äî two builder interactions are inherently pointer-only and are documented as such, not worked around: **parcel selection** (clicking a lot on the MapLibre canvas in step 2) and **freehand drawing** on the Konva surface. The complete keyboard-only route is the **upload path**: every control on it (plan-choice cards, file inputs, all form fields, the Draw/Type signature toggle and typed signature) is a native focusable element, and every builder step carries a focusable `[data-switch-upload]` bail-out. What *was* made keyboard-accessible: Orient-step rotation (arrow keys on the focused map container, see above) and the owner signature (Type method). When adding new interactions, keep the upload path's keyboard completeness intact.

## Gotchas

- **`EMAIL_TO` (app.js) CCs the developer:** it is `"carolmarie.taylor@fsresidential.com,steven@stevenbrown.design"`. The second address is the developer's, baked in for the demo.
- **Form-content edits go in the data constants** (`ACKS`/`PALETTE`/`TOOL_MODES`/`DATES`/`PHOTO_SPECS`), not the DOM, which is generated from them.
- **`Konva.toJSON()` returns a *string*, not an object.** `serializePlot()` does `JSON.parse(drawLayer.toJSON()).children` before nesting it into the draft object ‚Äî nesting the raw string produces double-escaped JSON bloat in `localStorage` (and in `renderPlotImage()`'s offscreen-stage rebuild).
- **Cross-module imports from `app.js` are function-only and event-time.** plot-editor.js and map-wizard.js import `scheduleAutosave`/`updateProgress`/`setFieldError`/`refreshPacketUI` back from the entry module (an intentional ESM cycle). Hoisted function declarations make this safe; importing a `const`/`let` from `app.js` and touching it at another module's top level throws a TDZ ReferenceError at page load. Conversely, the wizard's exported state (`planMode`, `parcelBearing`, `selectedAPN`, ‚Ä¶) is read cross-module as live bindings ‚Äî only map-wizard.js itself may assign them (use its setters: `setPlanMode`, `setRotation`, `restoreParcelFromDraft`).
- **A drag/click gesture that *releases* past the canvas edge never fires the Konva stage's own `pointerup`** ‚Äî native pointer events only reach elements the pointer is still over, so without a fallback, an in-progress shape/callout/measurement is silently abandoned instead of committed. `initPlotStage()` also binds `onStagePointerUp` on `window` as a fallback; the handler is idempotent (no-ops once state is already clear) so double-binding is safe.
- **Materials are raster, annotations are vector ‚Äî two separate persistence + undo paths.** The material grid lives in the `cellState` `Map` painted onto `paintCanvas` (so Fill is a *real* 4-way `floodFill`, not the old click-a-shape-to-recolor trick); callouts/measurements live as Konva nodes on `drawLayer`. `collect()`, `snapshotState()`/`applyHistorySnapshot()`, and `renderPlotImage()` each have to handle **both** ‚Äî forget one and either the painted grid or the annotations silently drops out of the draft / undo / print.
- **`[hidden]` vs author `display`:** hiding an element with the `hidden` attribute only works if no author rule sets `display` on it. Both `.layout` (`display:grid`) and `.modal` (`display:grid`) therefore need an explicit `.layout[hidden]`/`.modal[hidden] { display:none }` rule, and builder-only step dots use inline `style.display` instead of `hidden` (`.plot-steps-nav__dot` is `display:flex`). Same gotcha, three places.
- `ISSUES.md` is the running work log; **#3** (good-vs-bad photography examples) is now wired into the Photos section as **placeholder frames** ‚Äî the structure exists, the real comparison images are still pending.
- **MapLibre only runs one camera animation at a time.** The step 2‚Üí3 transition in `showStep()` zooms to the parcel via `mapInstance.fitBounds(...)` ~100ms after the step switches. Auto-align used to call a separate `mapInstance.easeTo({bearing})` for the rotation ‚Äî `fitBounds()` starting shortly after cancelled it mid-flight and froze the bearing whatever it had reached at that instant (usually still ~0¬∞). Fix: the target bearing is folded into the *same* `fitBounds({..., bearing: parcelBearing})` call instead of a standalone animation. If you ever add another camera-animating call near a `showStep()` transition, check whether it collides with an in-flight one the same way.
- **Never trust `e.features[0].geometry` from a MapLibre GeoJSON-source click handler for anything precision-sensitive.** GeoJSON sources are internally re-tiled (geojson-vt) for rendering, and the feature geometry returned by a click/`queryRenderedFeatures` is that simplified render geometry, not the original. The parcel-click handler resolves `selectedParcelGeoJSON` from `parcelsData` (the raw fetched GeoJSON) instead, re-fetching by APN via `fetchParcelGeometryByAPN()` if it's missing there ‚Äî this was the root cause of the Draw-step grid once rendering visibly narrower than the real parcel.
