# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A **static, client-side mockup** of the Fairway Canyon HOA Architectural Review Committee (ARC) application, rebuilt from a printed PDF (`docs/source-form.pdf`) as a web form for GitHub Pages. **There is no backend.** Nothing is transmitted: drafts persist to `localStorage`, "submitting" generates a printable preview or opens a `mailto:`, and data can be exported as JSON.

Plain HTML/CSS/JS — **no build step, no package.json, no npm dependencies, no tests, no linter.** The only external runtime dependencies are CDN `<script>`/`<link>` tags: MapLibre GL JS (maps), Konva.js (the plot-plan drawing stage — a raster material grid plus vector annotations), and Google Fonts. (There is one dev-time Node script, `tools/fetch-streets.mjs`, but it is not part of the app or its build — see Commands.)

## Commands

- **Run locally:** `python -m http.server 8000`, then visit `http://localhost:8000`. Use a server rather than opening `index.html` via `file://` — the county GIS `fetch()` calls and map tiles need an http(s) origin.
- **Deploy:** GitHub Pages serves the repo root (`.nojekyll` disables Jekyll processing). The live branch is **`master`**.
- **Sanity check:** there is no lint/test tooling, but always run `node --check assets/app.js` after editing it — it catches syntax errors before you hand a change to the user (who does the real browser verification). The app itself needs no build step: edit the three source files and reload.
- **Regenerate the baked street extract:** `node tools/fetch-streets.mjs` rewrites `assets/streets.json` from live Overpass (it tries several public mirrors in order). Widen the `BBOX` in that script if the service area grows. Dev-time tool only — the deployed app just `fetch()`es the committed JSON.

### Verification / testing this agent CAN'T do well

**Ask the user to test map-wizard and drawing-surface changes for you — don't burn turns trying to drive them via browser automation.** The map wizard (Select → Orient → Draw) and the Konva plot surface are effectively un-testable from Claude Code's browser automation in this environment, for concrete reasons learned the hard way:
- The MapLibre map frequently sits **below the fold and won't scroll into the clickable viewport**, so screen-clicking a parcel is unreliable.
- The dev server (`python -m http.server`) sends no cache headers, so the browser **serves a stale `assets/app.js` even after edits + reload** — you have to cache-bust the `<script src>` to load fresh code, which is throwaway churn.
- Driving the flow programmatically needs a **temporary `window.__dbg` hook** (private IIFE internals aren't otherwise reachable), plus the county GIS lookup, the auto-align camera animation, and the aerial snapshot capture all have real latency/timing that browser automation races against.
- Screenshots intermittently time out or render faded/blank in this sandbox.

So: implement the change, `node --check assets/app.js`, reason carefully about the geometry/timing, then **hand it to the user to verify in their own browser** (they have the real map, real parcels, and can eyeball scale/alignment instantly). Reserve heavy browser-automation verification for the static form sections that actually sit in the viewport.

## Architecture

Three files do everything: `index.html` (markup), `assets/styles.css` (presentation, ~1200 lines, organized in clearly-labeled `/* === SECTION === */` blocks ending with a `@media print` block that styles the print output), and `assets/app.js` (all behavior).

`assets/app.js` is a single ~3100-line IIFE. All state lives in module-level `let` variables; there are no modules or classes beyond `SignaturePad`. Content is data-driven from constants at the top of the file — **edit these arrays to change form content**, not the generated DOM:
- `ACKS` — the 8 acknowledgment statements (HTML strings, rendered into `#acks`)
- `PALETTE` — plot-plan materials (`id`/`label`/`color`), rendered into the material toolbar (`#palette`); `TOOL_MODES` is the separate list of drawing-tool modes (paint · rect · erase · fill · line · callout · measure · select · pan, each paired with an inline SVG from the `ICON` map), rendered into `#tool-palette`
- `DATES` — the 2026 review-date table (rendered into `#dates-body`, which lives on the **landing page**)
- `PHOTO_SPECS` / `PHOTO_MATERIAL` — the per-area photo requests for the Photos section (each shot has `id`/`title`/`instr`/`good`/`bad`); editing these changes which photos are requested and their instructions

The form is a **multi-section single page**, gated behind a **landing/cover screen**. DOM/section order (and card numbers): landing → `01` Applicant · `02` Site/Plot Plan · `03` Proposed Change · `04` Adjacent Owners · `05` Photos · `06` Required Submissions (confirm & attach) · `07` Acknowledgments. The sidenav, card numbers, and DOM order must agree when sections are added/reordered.

### Key subsystems in app.js

- **Landing / form reveal** — a full-screen cover (`#landing`, between the masthead and the form) holds the unpacked required-submissions overview + the 2026 review-dates table + how-to-submit/assistance. The whole form (`.layout`, `id="form-layout"`) starts with the `hidden` attribute; `enterForm()` (wired to `#start-application`) unhides it, and `showLanding()` (wired to the sidenav `#view-landing` link) returns to it. **Returning users skip the landing:** `restoreDraft()` returns a boolean and `enterForm()` is called when a draft was restored.

- **Photos** (`#photos-section`) — a **questionnaire drives which photos are requested**. The two questions (which areas: `area_front/back/side/exterior` checkboxes; plus a `photoMaterial` yes/no radio) toggle pre-built per-area groups via `refreshPhotoGroups()` (it flips each `.photo-group`'s `hidden`, and shows/hides the `#photo-empty` placeholder). Groups/blocks are generated once by `buildPhotoRequests()` from `PHOTO_SPECS`/`PHOTO_MATERIAL`, so already-attached files survive a questionnaire change (groups are hidden, not destroyed). Each shot has its own `<input type=file data-photo-input=ID>` + a ✓/✗ example pair (**placeholder frames** — real comparison images pending, ISSUES #3). `collect()` emits `photoAreas`, `photoMaterial`, and a `photos` map (`id → filename`); `files` is the flat list of attached photo names for preview/print. As with all file inputs, restore can only show the prior filename (browsers can't repopulate a file input), shown as an `is-prior` status.

- **`SignaturePad`** (the one class) — DPR-aware `<canvas>` signature capture via pointer events. Serializes to a PNG dataURL (`toDataURL`/`fromDataURL`) so signatures survive localStorage round-trips. Instances are tracked in the `sigPads` map, keyed by the element's `data-sigpad` attribute.

- **Plot drawing surface (raster grid painter + Konva annotations)** — the "Draw Your Plan" step (`#plot-konva-host`) is a `Konva.Stage` with **four** layers, bottom-to-top: `bgLayer` (`listening:false`; aerial snapshot + dashed parcel outline), `gridLayer` (`listening:false`; two stacked `Konva.Image`s wrapping offscreen `<canvas>`es — `gridLinesCanvas` graph paper + `paintCanvas` painted materials — with `mixBlendMode:"multiply"` set on the layer's own DOM canvas so materials read *through* the aerial on screen, while `toDataURL()` ignores the blend so print stays opaque), `drawLayer` (Konva callout + measurement groups + Line-tool lines **only**), and `overlayLayer` (in-progress ghost shapes + the Select-highlight rect — excluded from persistence/export just by living on its own layer). **Scale is fixed: 1 grid tile = 1 ft = `CELL_SIZE` (8) px at 100% zoom.** A bigger lot yields a *bigger grid you pan/zoom around*, never a shrink-to-fit one, so a foot is always the same size on screen. **Materials are raster**, held in the `cellState` `Map` (`"c,r" → materialId`) and drawn with real cell ops dispatched from `onStagePointerDown/Move/Up`: `paintBrush`/`paintStroke` (a square brush of `brushSize` ft, interpolated so a fast drag leaves no gaps), `fillRect` (Rectangle rubber-bands a block; **Shift** on Paint draws a straight line), and a genuine 4-way **`floodFill`** paint-bucket. Right-click / Ctrl(⌘)-click is a quick eraser from any tool. **Annotations are vector Konva**: callouts are a `Konva.Group` (`Label`+`Tag`+`Text` plus a free-angle `Konva.Arrow` leader, since `Tag.pointerDirection` only supports 4 cardinals) measurements a `Konva.Group` (`Konva.Arrow` + a `Konva.Text` distance label via `scaleFeetPerPixel`), and the **Line** tool a plain `Konva.Line` (`kind:"line"`) whose endpoints `snapToGrid` to the nearest grid-line intersection and whose stroke is a real-world 2 in (`gridLineWidth()` = `LINE_WIDTH_FEET / scaleFeetPerPixel`, so it stays 2 in at any zoom) with a fat `hitStrokeWidth` so the hairline is still clickable — and Lines also act as **flood-fill walls at any angle** (triangles included): `floodFill` floods at `FILL_SUBSAMPLE`² (4×4) sub-tile resolution and blocks a step whenever the connector between two subcell centres crosses a Line (`collectLineSegments`/`segmentsIntersect`/`connectorBlocked`) — watertight for straight segments since a straight connector crosses a straight segment iff its endpoints are on opposite sides, so a closed shape whose corners share snapped vertices fully contains the fill — then paints a whole tile iff **>50% of its subcells are inside** (the "more than half the tile is within bounds" rule, which also subsumes axis-aligned walls and 100%-interior tiles). Coverage = reached subcells **plus the on-wall strip** (`pointOnAnyWall`) that the conservative touch-block otherwise erodes — without this a diagonal-cut tile reads <50% and leaves a visible gap, so triangles wouldn't fill to their edge. Filling the *other* side of a wall can't overwrite already-painted edge tiles (the material-region check blocks the second flood from entering them), so the **first fill keeps the shared edge — no gap, no overwrite**. The fill is seeded from the subcell under the click so it starts on the clicked side of a wall; `attachShapeInteractions` wires Select (drag-to-move) and Erase (click-to-delete) per node, and annotations stay `listening(false)` outside those two modes so a paint stroke passes straight through to the grid beneath. **Pan/zoom is DOM-scroll based**: `applyStageSize` scales the stage by `viewZoom` and the `#plot-konva-host` element scrolls; `minZoom`/`maxZoom`/`clampZoom` bound you between "no whitespace" and a `SAFE_CANVAS_PX` memory cap; `fitView` ("Fit") zooms out and centers on the parcel; the Pan tool / middle-mouse drag adjusts native scroll. **Undo/redo (`snapshotState`) captures BOTH** `cellState` (as pairs) **and** `drawLayer.toObject().children` onto a capped (50-deep) stack, recorded via `recordUndoPoint()` at the *start* of each gesture, not the end. `renderPlotImage()` (used by `buildPreview()`/`buildPrintHTML()`) composites a **throwaway offscreen `Konva.Stage`** — never the live one — from `gridLinesCanvas` + `paintCanvas` + the parcel outline + `drawLayer.toJSON()`, so pan/zoom state and overlay ghosts can never leak into print. Autosave (`scheduleAutosave()`) is called explicitly from every tool's commit handler, since canvas gestures never fire the native `input` event. A `#plot-fallback-msg` shows if Konva fails to load (`KONVA_AVAILABLE`).

- **Parcel → grid projection** — a county GeoJSON parcel polygon becomes pixel space via: `geoToLocalMeters` (lng/lat → local meters) → `rotatePoints` (by `-bearing` to match the rotated map view) → flip Y (geographic north-up → screen rows-down) → `computeBBox`. **The grid is fixed-scale (1 tile = 1 ft), not fit-to-viewport**: `buildParcelGrid` sizes a *square* whose side is the parcel's longest dimension + ~10% buffer (`GRID_MARGIN`, at least `GRID_MIN_PAD` tiles, clamped to `MAX_GRID_DIM`), centers the parcel in it, and returns `{cols, rows, polygonPx, metersPerCell}` — **no per-cell `mask` anymore** (the footprint is traced as a dashed `Konva.Line`, not cell-clipped). `metersPerCell` (= `FOOT_IN_METERS`) feeds `scaleFeetPerPixel = (metersPerCell * 3.28084) / CELL_SIZE`, the real-world scale the Measure tool uses. **`gridCols`/`gridRows` are mutable tile counts** (= feet) recomputed per parcel; `rebuildGridForParcel()` rebuilds the grid canvases + `bgLayer` and clears both `cellState` and `drawLayer` — **but re-entering the Draw step with the same parcel/bearing just re-fits the view without wiping** (guarded by `lastPlotAPN`/`lastPlotBearing`; a `confirm()` fires only when the parcel/orientation actually changed under an already-drawn plan). The aerial backdrop is registered to the outline by a least-squares similarity fit (`fitSimilarity`) over corresponding parcel vertices, then shifted by a baked ground-vector ortho correction (`AERIAL_NUDGE_FT`, rotated by bearing into stage space) — the residual ~few-foot orthophoto offset is a verified *non*-datum issue (a NAD83↔WGS84 transform moves the parcels 0.00 ft), so it's tuned by eye here rather than reprojected away.

- **Map wizard** — the Site/Plot Plan section (`#siteplan`) is a 4-step flow driven by `showStep(n)`: **1 Method → 2 Select → 3 Orient → 4 Draw**. Step 1 is a `planMode` choice (`"build"` vs `"upload"`): "Build a plan" runs `startBuilder()` (which reads `#property-address` from the standalone Applicant Information card and kicks off geocoding) and reveals the Select/Orient/Draw dots; "Upload an existing plan" reveals a file input (`#plot-upload`) and hides those dots. Map data comes from MapLibre GL + OpenFreeMap tiles + **Riverside County ArcGIS REST** services: the assessor table (layer `50`) resolves address → APN, and the parcel layer (`40`) returns polygon geometry. Note: the **single `#map-container` DOM node is physically moved** between the step-2 and step-3 containers via `target.prepend(mapContainer)` — there is only one map instance, reparented as you navigate. Step 2 locks north-up; step 3 enables rotation. **Orientation is drag-to-rotate directly on the map** (mousedown + horizontal drag at `ROTATE_DEG_PER_PX` = 0.5°/px, active in step 3 only) plus **±0.1° fine-nudge buttons** (`#rotate-fine-minus`/`-plus` → `nudgeRotation()`, snapping to the nearest 0.1°) with a live tenths-of-a-degree readout (`#map-rotate-fine`); the old slider / **±15°** / Reset / Auto-align panel is commented out in both `index.html` and `app.js`. Both the nudge buttons and drag live *inside* `#map-container` so they ride along when the node is reparented, and the fine control is revealed only by the `#map-step3 .rotate-fine` rule (same pattern as the orient overlay). Rotation funnels through a single `setRotation(deg)` function (normalizes mod 360 **without rounding, so fractional/tenths degrees survive**, updates `parcelBearing`, `syncRotationReadout`, and `mapInstance.setBearing`) — the drag handler, the nudge buttons, `restoreDraft()`, and `computeAutoAlignBearing()` (which squares the view to the parcel's longest boundary edge **and then picks the front-yard-down flip of the two 180°-apart candidates** using `selectedParcelRoadDir` — a unit vector toward the nearest street; falls back to the legacy `[0,180)` tie-break when that's unavailable; **Auto-align** no-ops without a `selectedParcelGeoJSON`) all funnel through it. **Nearest-street lookup is local-first:** `localNearestRoadDir()` reads a pre-baked OSM extract (`assets/streets.json`, loaded once into `streetData` via `streetDataReady`) and computes the direction in-memory with **zero network latency** for any parcel inside its bbox; only addresses *outside* that box fall through to `fetchNearestRoadDir()` → **Overpass (live OSM) over plain REST** (multi-mirror with a 7s per-endpoint `AbortController` timeout in `overpassQuery`). The lookup is kicked off in the parcel-click handler the moment the boundary resolves (Step 2) and cached in `selectedParcelRoadDir`; the Step 2→3 auto-align and the Auto-align button both `roadDirReady.then(...)`-wait for it (the 2→3 path caps the wait at 2.5s, and re-applies a late-arriving correction unless `userAdjustedRotation`). Why OSM and not the obvious sources: the county centerline layer (`OpenData/Transportation/0`) is **missing Fairway Canyon's residential streets entirely** (its "AARON" segments are 16+ km away — a different road), and the basemap's own OpenMapTiles `transportation` vector tiles come back empty from `querySourceFeatures` at align time (tile load/visibility timing), so neither is reliable here. **`assets/streets.json`** is a one-time Overpass extract (`way[highway=…]` within bbox `[-117.085,33.925,-117.01,33.985]`, ~901 streets, ~300 KB, © OpenStreetMap contributors/ODbL); regenerate it with a bounded Overpass `out geom;` query if the service area changes. Note the loader `fetch()`es it same-origin, so it only loads over http(s) (`python -m http.server`), **not** `file://` — under `file://` it silently falls back to live Overpass. Step 3 also dismisses the "Selected parcel" popup (`selectedPopup`, created on parcel click) on entry, and shows a transparent **Back Yard / Front Yard overlay** (`.map-orient-overlay`) — that overlay lives *inside* `#map-container` so it rides along when the node is reparented, but is `display:none` by default and only revealed by the `#map-step3 .map-orient-overlay` rule. The map's zoom/compass control buttons are likewise hidden in step 3 only via `#map-step3 .maplibregl-ctrl-top-right { display:none }` (kept for step 2's parcel selection). Gotcha: builder-only step dots are hidden via inline `style.display` (not the `hidden` attribute), because `.plot-steps-nav__dot` sets `display:flex`, which would override `[hidden]`.

- **Address geocoding** — `parseAddress()` regex-splits "number + street name" and queries the county assessor endpoint. It is **hardcoded to `CITY='BEAUMONT'`** and Riverside County URLs — this app is not geographically portable without changing those endpoints and the city filter.

- **Persistence** — `collect()` serializes the entire form (text fields, checkboxes, neighbors, all signatures as dataURLs, and the plot) into one object. The **plot is two things now**: `plot: {version:3, cell, cols, rows, cells: [...cellState], annotations}` — the raster grid as `Map` pairs **plus** the vector annotations (`JSON.parse(drawLayer.toJSON()).children`) — and `plotMeta` with parcel coords/APN/bearing. `saveDraft`/`restoreDraft` read/write `localStorage` under **`DRAFT_KEY = "fairwayCanyonArcDraft.v4"`**. Autosave is debounced 1200ms via `scheduleAutosave()`, called both from the form's native `input` listener and directly from every canvas tool's commit handler. Restoring rebuilds the parcel geometry from `parcelCoords` (`rebuildGridForParcel`), repaints the grid via `loadCells(plot.cells)`, then hydrates annotations via `hydrateShapesInto` → `Konva.Node.create` + `attachShapeInteractions` (serialized Konva JSON carries no listeners, so skipping the re-attach leaves shapes visible but inert). If no `.v4` draft exists, `restoreDraft()` walks **`LEGACY_DRAFT_KEYS = [".v3", ".v2"]` once**: every non-plot field migrates losslessly, but the drawing is **left empty** — neither the old Konva-vector `.v3` plan nor the pre-Konva `.v2` cell-grid maps onto the new fixed-scale grid — with a one-time notice (`#plot-migration-notice`), and the migrated draft is immediately re-saved under `.v4` so the migration runs once per browser.

- **Output paths** — four ways data leaves the form, all client-side: the **preview modal** (`buildPreview`), a **single-page print window** (`buildPrintHTML` → `printPreview` opens `window.open` and writes a self-contained doc with an inline `<style>` block — print CSS lives there, not in styles.css), a **`mailto:`** link (`openMailto`), and **JSON download**. `collect()` is the single source feeding all four. There is also a fifth, separate print window: `printNeighborForm()`/`buildNeighborFormHTML()` generates the **wet-signature form for adjacent owners** (the Adjacent Owners section collects names/addresses digitally, but signatures are gathered on this printout and re-attached via `#neighbor-form-file`).

- **Validation / progress** — `validate()` gates submit/preview: it walks required fields, applies `PHONE_RE`/`EMAIL_RE` format checks, and paints per-field messages via `setFieldError` (cleared on input). `updateProgress()` drives the sidenav completion meter from the same `[required]` census plus the owner acknowledgment signature.

## Gotchas

- **`EMAIL_TO` (app.js) CCs the developer:** it is `"carolmarie.taylor@fsresidential.com,steven@stevenbrown.design"`. The second address is the developer's, baked in for the demo.
- **Form-content edits go in the data constants** (`ACKS`/`PALETTE`/`TOOL_MODES`/`DATES`/`PHOTO_SPECS`), not the DOM, which is generated from them.
- **`Konva.toJSON()` returns a *string*, not an object.** `collect()` does `JSON.parse(drawLayer.toJSON()).children` before nesting it into the draft object — nesting the raw string produces double-escaped JSON bloat in `localStorage` (and in `renderPlotImage()`'s offscreen-stage rebuild).
- **A drag/click gesture that *releases* past the canvas edge never fires the Konva stage's own `pointerup`** — native pointer events only reach elements the pointer is still over, so without a fallback, an in-progress shape/callout/measurement is silently abandoned instead of committed. `initPlotStage()` also binds `onStagePointerUp` on `window` as a fallback; the handler is idempotent (no-ops once state is already clear) so double-binding is safe.
- **Materials are raster, annotations are vector — two separate persistence + undo paths.** The material grid lives in the `cellState` `Map` painted onto `paintCanvas` (so Fill is a *real* 4-way `floodFill`, not the old click-a-shape-to-recolor trick); callouts/measurements live as Konva nodes on `drawLayer`. `collect()`, `snapshotState()`/`applyHistorySnapshot()`, and `renderPlotImage()` each have to handle **both** — forget one and either the painted grid or the annotations silently drops out of the draft / undo / print.
- **`[hidden]` vs author `display`:** hiding an element with the `hidden` attribute only works if no author rule sets `display` on it. Both `.layout` (`display:grid`) and `.modal` (`display:grid`) therefore need an explicit `.layout[hidden]`/`.modal[hidden] { display:none }` rule, and builder-only step dots use inline `style.display` instead of `hidden` (`.plot-steps-nav__dot` is `display:flex`). Same gotcha, three places.
- `ISSUES.md` is the running work log; **#3** (good-vs-bad photography examples) is now wired into the Photos section as **placeholder frames** — the structure exists, the real comparison images are still pending.
- **MapLibre only runs one camera animation at a time.** The step 2→3 transition in `showStep()` zooms to the parcel via `mapInstance.fitBounds(...)` ~100ms after the step switches. Auto-align used to call a separate `mapInstance.easeTo({bearing})` for the rotation — `fitBounds()` starting shortly after cancelled it mid-flight and froze the bearing whatever it had reached at that instant (usually still ~0°). Fix: the target bearing is folded into the *same* `fitBounds({..., bearing: parcelBearing})` call instead of a standalone animation. If you ever add another camera-animating call near a `showStep()` transition, check whether it collides with an in-flight one the same way.
- **Never trust `e.features[0].geometry` from a MapLibre GeoJSON-source click handler for anything precision-sensitive.** GeoJSON sources are internally re-tiled (geojson-vt) for rendering, and the feature geometry returned by a click/`queryRenderedFeatures` is that simplified render geometry, not the original. The parcel-click handler resolves `selectedParcelGeoJSON` from `parcelsData` (the raw fetched GeoJSON) instead, re-fetching by APN via `fetchParcelGeometryByAPN()` if it's missing there — this was the root cause of the Draw-step grid once rendering visibly narrower than the real parcel.
