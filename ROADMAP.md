# Roadmap

Distilled from a full-project critique (2026-07-02): architecture, robustness, and user-workflow reviews. Work proceeds **one sprint at a time** — see the working agreement in `CLAUDE.md`. Each sprint is sized to roughly one working session. Don't renumber sprints; mark items done (`~~strikethrough~~` or ✅) and log outcomes in `ISSUES.md`.

**The through-line:** the form is excellent at *collecting* (landing, wizard, photo questionnaire) but treats *assembling and delivering the packet* as the user's problem — and the HOA's own rules make incompleteness unforgiving (auto-denial, 45-day clock never starts). Sprints 1–2 fix that. Sprints 3–5 pay down robustness, code-health, and accessibility debt.

## Sprint 13 — Eraser & object lifecycle (items 11–13) ✅ 2026-07-04 (+ same-session extras from the 2026-07-04 evening request list)

Erasing was split-brained: right-click / Ctrl+click quick-erase only nulled **grid cells** from any tool, while vector objects (stamps, lines, callouts, measurements) died only to an Erase-mode left click — and annotations are `listening(false)` outside Erase/Select, so a right-click on a stamp erased the paint *under* it instead. There was also no keyboard delete.

- [x] **One erase resolution, topmost-first** — deletion funnels through one `deleteAnnotation()` (undo point → drop selection if it was the selected node → destroy → cursor reset → `updateProgress()` + autosave). ## Sprint 23 — Aerial/parcel registration accuracy (two-phase fix) 🔶 IN PROGRESS 2026-07-05

**Origin (2026-07-05/06 conversation, demo-mode follow-on):** while baking a real address into the demo-mode toggle (see the "Demo mode" ISSUES.md entry), the user spotted the aerial photo badly misaligned with the parcel outline on a Gallery Ln lot — which turned out to be a **double bug**: (a) that address is in **Tournament Hills**, a *different* Beaumont gated community, not Fairway Canyon at all (`assets/parcels.json`'s bbox is generously sized around the streets/parcels extract and evidently spans multiple unrelated subdivisions — it is NOT an HOA-boundary-filtered extract, just a geographic box); and (b) even after that's accounted for, the user found the drift "objectionable" on multiple *bona fide* Fairway Canyon lots while browsing around the map for a replacement address. This sprint is about (b) — the real registration-accuracy problem — plus the quick fix for (a).

**Diagnosis (reasoned through in the 2026-07-05 conversation, not yet verified in code):** `buildAerialNode()` (`plot-editor.js`, ~L881) registers the fetched aerial photo to the drawn parcel outline via `fitSimilarity()` (`geometry.js` L59) — a least-squares rigid fit (rotation + uniform scale + translation) between two point sets that come from **two different map projections**: the grid's own parcel outline (`parcelPolygonPx`, from `buildParcelGrid()` in `geometry.js`, which projects via a local flat-earth tangent plane / `geoToLocalMeters()` so "1 cell = 1 real foot" holds exactly) vs. the aerial image's parcel-vertex pixels (`plotBgParcelPx`, from `fetchAerialBackdrop()` in `map-wizard.js`, which projects via true Web Mercator / `lngLatToMercator()`, since that's what the county ImageServer's `bboxSR=3857` export expects). Any two distinct projections agree only approximately over a small area, and — this is the key mechanism — a least-squares similarity fit is anchored at the point-set centroid, so whatever small residual exists between the two projections is **linearly amplified with distance from that centroid**. A tiny (sub-percent) residual is invisible on a small, compact, roughly-square lot; the same residual is very visible at the far vertices of a bigger or more irregularly-shaped lot (e.g. a pie-shaped cul-de-sac parcel). On top of this per-parcel dynamic fit, a single constant `AERIAL_NUDGE_FT = { east: -1.0, north: -3.6 }` (`plot-editor.js` ~L334, comment: "tuned by eye over two passes") is added — necessarily calibrated against whatever one reference case was on-screen at the time, so it doesn't generalize to every lot's geometry.
